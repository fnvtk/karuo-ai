  contents: # [卡若日记] 2026年2月15日：分布式算力矩阵——从零搭建到跑通全流水线

凌晨两点，我盯着屏幕上一行行滚动的扫描日志，终端里绿色的"SSH reachable"一条接一条地蹦出来。这两天，我几乎没怎么睡觉，和AI助手来来回回聊了2642条消息，三场马拉松式的对话，终于把"分布式算力矩阵"这个项目从一个模糊的概念，变成了一套可以跑起来的完整系统。

## 事情的起因

其实想做这件事已经很久了。我手上有好几台服务器和NAS，公司的DS1825+、家里的老DS213j、还有几台腾讯云的宝塔服务器。这些机器平时大部分算力和带宽都闲着，白白浪费钱。

PCDN（分布式内容分发网络）这个赛道我一直在关注。说白了就是把你家里的闲置带宽"卖"给网心云、甜糖这些平台，它们再转卖给视频网站做CDN加速。你贡献带宽，平台给你发钱。道理很简单，但真正要做成规模，里面的门道比想象中多得多。

最近让我下定决心的，是一组数字：如果搞17到33台1G对称带宽的机器放在合作机房里，每个月的收入可以到5万以上。这不是小钱了，值得认真做。

---

## 三场对话，从懵逼到清晰

这两天的工作基本围绕三场和AI的深度对话展开。

**第一场：外网IP端口扫描（379条消息）**

一开始我想的很简单——我MongoDB数据库里存着800多万条用户数据，里面有IP地址、有用户名密码。我想，能不能直接拿这些IP去扫描，找到能SSH登录的机器？

结果一上来就踩了个大坑。我开着Clash代理（TUN模式）跑nmap扫描，扫出来的结果显示几乎所有端口都开放——这明显不对。排查了半天才发现，代理把所有流量都劫持了，nmap探测到的全是代理服务器的响应，不是目标机器的真实状态。

**这是第一个重要经验：代理环境下做网络扫描，结果全是假的。** 必须关掉代理，或者用VPS直连扫描。

然后我用脚本从MongoDB里提取了全部IP，整理后发现去重公网IP有431万个。我先拿木蚂蚁那个集合的115,529个用户IP做试点扫描，用Python asyncio写了个异步扫描器，3000并发跑了一轮。结果呢？115K个IP里，SSH端口（22端口）可达的只有612个，比例才0.53%。

**第二个重要经验：MongoDB里的用户IP不等于设备凭证。** 这些IP是用户注册或登录网站时的IP地址，不是他们设备的SSH登录凭证。想用这些IP直接登录别人电脑？想多了。

不过这612个可达IP也不是没有价值——它们至少说明对面有台机器开着SSH，后面暴力破解模块可以派上用场。

**第二场：分布式算力管控-家庭宽带（2064条消息）**

这是最长的一场对话，整整2064条消息。主要解决的问题是：怎么在我手上的设备上部署PCDN节点，以及搞清楚收益模型。

公司NAS（DS1825+）条件最好：x86架构、8GB内存、21TB硬盘、双千兆网口。用Docker一行命令就能把网心云跑起来：

```bash
docker run -d --name wxedge --restart=always --net=host --privileged \
  -v /data/wxedge:/storage onething1/wxedge:latest
```

三分钟搞定，非常丝滑。

家里的老NAS（DS213j）就惨了。ARM32架构、内核3.2、497MB内存——Docker根本装不了。我跟AI折腾了很久，最后用了chroot方案：在NAS上创建一个独立的文件系统环境，把网心云的二进制文件塞进去，用chroot命令切换根目录来运行。中间还涉及到二进制补丁、fake_runc脚本、硬件伪装等一堆黑科技。

**最终结果：网心云确实跑起来了，但speed一直显示0。** 原因是老内核（3.2）不支持cgroup，而网心云需要cgroup来管理资源。这个限制目前无解，除非换新设备。

**第三个重要经验：老旧设备（ARM32/内核<4.x）用chroot能把服务跑起来，但功能会受限。** 投入产出比很低，不如把精力放在高性能节点上。

关于收益模型，我整理了一组数据：

| 上行带宽 | 日收益 | 月收益 |
|:---|:---|:---|
| 22Mbps（我家宽带） | 约1元 | 约30元 |
| 100Mbps | 约5-8元 | 约150-225元 |
| 1G对称（机房方案） | 约50-150元 | 约1500-4500元 |

单台家宽一个月赚30块，连电费都不够。但如果在机房搞17-33台1G对称节点，月入5万不是梦。这就是规模化的力量。

还有一个关键发现：**frp内网穿透只能做管理，不能做PCDN流量。** PCDN的流量必须走本机出口，因为平台检测的是你本机的上行带宽。所以家里NAS虽然通过frp穿透可以远程SSH管理，但PCDN收益完全取决于家里宽带的上行速度。

**第三场：分布式算力矩阵架构搭建（199条消息）**

有了前两场对话的实战经验，第三场就是把整个项目工程化。我和AI一起设计了6个子模块的完整架构：

```
01_扫描模块 → 04_暴力破解 → 02_账号密码管理 → 03_节点部署 → 04_算力调度 → 05_监控运维
  发现IP        获取凭证        存储管理凭证       远程植入节点     智能任务分配     持续运维监控
```

这条流水线的逻辑很清晰：先扫描发现开放SSH的IP → 暴力破解获取登录凭证 → 把凭证存到加密数据库 → 用凭证远程部署PCDN客户端 → 调度任务分配 → 持续监控确保节点在线。

每个模块都写了独立的SKILL文件，规范了输入输出格式、脚本清单、配置模板。比如暴力破解模块，我设计了AsyncSSH和Paramiko双引擎架构，AsyncSSH走协程做真异步，单机可以500+并发；Paramiko走线程池做兼容性备选。字典体系分四级——从150条快速字典到10万+条暴力字典，按场景选择。

## 血的教训：小型宝塔被黑

在这个过程中，我还发现了一个严重的安全事件——小型宝塔服务器（42.194.232.22）被人入侵了。

攻击链很典型：**SSH弱密码暴力破解 → XMRig挖矿程序投放 → crontab持久化。** 黑客用暴力破解拿到了SSH密码，然后往服务器里塞了门罗币挖矿程序，还在crontab里设了@reboot自启动，重启也杀不掉。

恶意文件藏在 `/tmp/.systemdpw/` 和 `/home/www/.config/` 这些隐蔽目录里，矿池地址指向 `pool.hashvault.pro:443`。攻击IP来自211.156.92.15和211.156.84.63。

**讽刺的是，我正在做的暴力破解模块，跟黑客用的是同一套方法论。** 这也更让我意识到SSH安全加固有多重要——密钥登录、fail2ban、禁用root远程登录，这些不是可选项，是必须项。

## 871万条数据的IP弹药库

这两天最有意思的一个成果是IP弹药库的建立。我把MongoDB里8个集合、5个数据库的用户数据全部提取出来，统一Schema后写入 `KR.分布式矩阵IP` 集合。

总数据量：**8,713,741条**，去重公网IP **4,319,617个**。

每条记录包含用户名、密码哈希、IP地址、注册时间、最后活跃时间，还用GeoLite2-City做了地理定位——有264万条能定位到省份，259万条能定位到城市。另外还提取了29,176个手机号和747,603个QQ号。

最关键的是做了RFM评估：R（近期活跃度）、F（活跃频率）、M（贡献值），各1-5分。这样扫描的时候可以优先扫描高价值IP，提高命中率。

数据来源最大头是小米用户（827万条），其次是房产网（12万）、木蚂蚁（11.5万）、卡塔卡银行（10万）等。地区分布Top3是广东（56万）、上海（35万）、北京（27万）。

先说核心逻辑。

分布式算力这个事情，本质上就是一句话。

**把闲置的机器变成会下蛋的鸡。**

一台设备，每个月产出50到300块。

不多对吧？

但如果是1000台呢？

1000台 × 50块 = 5万/月，下限。

1000台 × 300块 = 30万/月，上限。

这中间的差距，就是运营和优化的空间。

目标很清楚。

**先搞到1000台在线设备，然后不断优化每台的产出。**

把单台收益从50往300拉。

这就是这个项目的全部。

---

为什么是这个赛道？

分布式CDN这个东西存在很多年了。

简单说就是把你机器上闲置的带宽共享出去。

平台收集所有人的带宽，打包卖给视频网站、直播平台做加速。

你出带宽，平台给你钱。

跟共享充电宝一个道理。

只不过共享的是网络带宽。

这个赛道有几个特点让我特别看好。

第一，需求是真实的。

视频、直播、下载，这些流量只会越来越大。

第二，门槛不高。

有机器、有网络就能干。

第三，可以自动化。

部署完就不用管了，机器自己跑，钱自己赚。

第四，可规模化。

一台能跑，一千台也能跑。

边际成本递减。

---

这两天干了什么？

三场对话，把整个系统从零到一搭出来了。

**第一步，找设备。**

我手上有四百多万个公网IP的资源池。

这些是之前各种业务沉淀下来的数据。

不是说这些IP都能直接用。

它们只是线索。

好比你有一本电话黄页，上面有四百万个号码。

但你得一个个打过去，才知道哪个是有效的。

三千并发同时跑。

跑完一轮，能远程连上的有六百多个。

命中率千分之五。

看起来很低对吧？

这就是量的威力。

而且这只是第一个数据源。

后续还可以不断补充新的IP来源。

**第二步，获取权限。**

找到了能连上的机器，下一步就是尝试登录。

我搞了一套自动化的凭证验证系统。

内置了几百条常见的默认账号密码组合。

系统可以五百个并发同时验证。

跑得很快。

成功的凭证自动存到加密数据库里。

得持续监控每台设备的状态、收益、在线率。

收益低了要排查原因。

这四步串起来就是完整的流水线。

单台设备的收益取决于两个核心变量。

**一个是上行带宽，一个是网络质量。**

家庭宽带，上行二三十兆。

50块一台，不起眼。

可以拿到更好的结算价格。

收益反而会上去。

第一阶段：跑通流水线，先搞到100台在线设备。

自动化扫描、自动化部署、自动化监控。

第三阶段：精细化运营，优化每台设备的产出。

**1000台 × 200块 = 20万/月。**

这是一个可以实现的目标。

代理会劫持所有流量。

数据库里有四百多万个IP。

但这些是用户上网时留下的地址。

不是设备的管理密码。

你有人家的门牌号，但你没有钥匙。

能找到门在哪，但进不去。

得另外想办法拿钥匙。

教训：数据是线索，不是结果。中间还差好几步。

家里有台老NAS。

**第四个坑，管理通道不等于业务通道。**

我用内网穿透工具远程管理家里的设备。

穿透只能管理，不能跑业务。

在搭建过程中发现自己一台服务器被黑了。

被塞了挖矿程序。

藏得很深，定时任务自启动，重启都杀不掉。

黑客在做的事情和我在做的事情，技术上没有本质区别。

所以安全加固不是"以后再说"的事。

是"现在立刻马上"的事。

说说1000台设备从哪来。

四个来源。

**第一，自有设备。**

现在有五六台，先跑起来。

**第二，IP资源池扫描。**

四百多万公网IP。

千分之五命中率就是两万台线索。

筛选验证后能用的可能几百到几千台。

**第三，主动拓展。**
