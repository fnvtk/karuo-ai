# 需求拆解与计划制定

> 先思考，再拆解，后执行。结合卡火（火）的深度思考与卡人（水）的任务规划能力。

## 核心理念

### 三大原则

1. **思考优先**：每次对话前先深度思考需求本质
2. **拆解为要**：将复杂需求拆解成可执行的原子任务
3. **执行跟踪**：按计划执行，持续更新状态

### 协作分工

| 阶段 | 负责人 | 能力 |

## 第一阶段：深度思考（卡火）

### 1.1 每次对话前先思考

**规则**：在开始任何任务前，先进行3-5分钟的深度思考。

**思考框架**：

```markdown
## 🧠 需求思考

### 1. 需求本质（5 Whys）
- 用户真正想要什么？
- 为什么需要这个？
- 不做的后果是什么？

### 2. CRITIC审视
- **C澄清**：需求的具体含义是什么？
- **R假设**：有哪些隐藏假设？
- **I证据**：有数据或案例支持吗？
- **T视角**：不同角色会怎么看？
- **I后果**：做/不做的后果是什么？
- **C原点**：这真的是核心问题吗？

### 3. Human 3.0追问（如需要）
- 目标澄清：解决什么问题？给谁用？
- 边界定义：做什么？不做什么？
- 资源约束：时间？预算？人力？
- 异常处理：失败了怎么办？
- 验收标准：怎么测试？

### 4. 思考结论
- 核心目标：
- 关键约束：
- 风险点：
- 下一步：
```

### 1.2 CRITIC思考模板

| 维度 | 问题 | 输出 |
|:---|:---|:---|
| **C澄清** | "这个需求具体指什么？范围在哪？" | 明确需求边界 |
| **R假设** | "有哪些隐藏假设？这些假设成立吗？" | 识别假设风险 |
| **I证据** | "有数据/案例支持吗？来源可靠吗？" | 验证需求合理性 |
| **T视角** | "用户/开发者/运营会怎么看？" | 多角度审视 |
| **I后果** | "做/不做的短期和长期影响？" | 评估影响 |
| **C原点** | "这真的是最重要的问题吗？" | 回归本质 |

### 1.3 5 Whys深挖技巧

**示例**：

```
用户："我想要一个导出功能"
→ Why 1: 为什么需要导出？
用户："为了做周报"
→ Why 2: 为什么要手动做周报？
用户："因为需要给老板看数据"
→ Why 3: 为什么不能自动生成？
用户："因为数据分散在不同系统"
→ Why 4: 为什么不能统一数据源？
用户："因为历史原因，系统是分散的"
→ Why 5: 为什么不能做数据同步？
→ 本质需求：数据统一 + 自动化报表，而非简单的导出功能
```

## 第二阶段：需求拆解（卡人）

### 2.1 拆解原则

1. **原子化**：每个任务30分钟~4小时可完成
2. **可测试**：每个任务有明确的验收标准
3. **依赖清晰**：明确写出前置任务
4. **优先级明确**：P0核心/P1重要/P2可选

### 2.2 拆解步骤

```
1. 提取功能点
   PRD/需求 → 功能模块 → 功能点

2. 识别依赖关系
   数据库设计 → API开发 → 前端对接 → 测试

3. 设置优先级
   P0: 核心功能，必须完成
   P1: 重要功能，应该完成
   P2: 锦上添花，可以延后

4. 估算时间
   单个任务：30m / 1h / 2h / 4h
   整个项目：按任务累加 + 20%缓冲

5. 生成任务清单
   格式：Epic → Task → Subtask
```

### 2.3 任务粒度标准

| 粒度 | 时间 | 示例 |
|:---|:---|:---|
| **Epic** | 1-2周 | "用户登录模块" |
| **Task** | 4-8小时 | "实现微信登录" |
| **Subtask** | 30分钟-2小时 | "调用微信API获取code" |

3. **持续思考**：遇到问题时重新思考（调用卡火）

   - 调整依赖关系
   - 重新安排优先级

## 拆解原则

### 1. 原子化原则

**标准**：每个任务30分钟~4小时可完成

| ✅ 好的拆解 | ❌ 差的拆解 |
|:---|:---|
| T1.1: 调用微信API获取code（30m） | T1: 实现微信登录（2天） |
| T1.2: 后端验证code并生成token（1h） | |
| T1.3: 前端存储token并跳转（30m） | |

### 2. 可测试原则

每个任务必须有明确的验收标准：

```
任务：T1.1 调用微信API获取code
验收标准：
- [ ] 能成功调用微信API
- [ ] 正确获取code参数
- [ ] 处理API异常情况
```

### 3. 依赖清晰原则

明确写出前置任务：

```
T1.2 依赖 T1.1
T1.3 依赖 T1.1, T1.2
```

### 4. 优先级明确原则

| 优先级 | 含义 | 示例 |
|:---|:---|:---|
| **P0** | 核心功能，必须完成 | 用户登录、支付 |
| **P1** | 重要功能，应该完成 | 数据统计、导出 |
| **P2** | 锦上添花，可以延后 | 主题切换、动画效果 |

---

## 拆解步骤

### Step 1: 提取功能点

```
需求文档
  ↓
功能模块（Epic）
  ↓
功能点（Task）
  ↓
子任务（Subtask）
```

**示例**：

```
需求：用户登录系统

Epic 1: 微信登录
  ├─ Task 1.1: 前端获取微信授权
  │   ├─ Subtask 1.1.1: 调用微信JS-SDK
  │   └─ Subtask 1.1.2: 处理授权回调
  ├─ Task 1.2: 后端验证并生成token
  │   ├─ Subtask 1.2.1: 调用微信API验证code
  │   └─ Subtask 1.2.2: 生成JWT token
  └─ Task 1.3: 前端存储token
      └─ Subtask 1.3.1: 存储到localStorage
```

### Step 2: 识别依赖关系

**依赖类型**：

| 类型 | 说明 | 示例 |
|:---|:---|:---|
| **技术依赖** | 需要先完成技术基础 | 数据库设计 → API开发 |
| **功能依赖** | 功能之间的依赖 | 登录 → 个人中心 |
| **数据依赖** | 需要先有数据 | 用户数据 → 数据统计 |

**依赖图示例**：

```
数据库设计
    ↓
API开发
    ↓
前端对接
    ↓
测试
```

### Step 3: 设置优先级

**优先级判断标准**：

1. **P0核心功能**：
   - 没有这个功能，系统无法运行
   - 用户核心路径必须的功能

2. **P1重要功能**：
   - 提升用户体验
   - 提高系统效率

3. **P2可选功能**：
   - 锦上添花
   - 可以延后实现

### Step 4: 估算时间

**时间估算原则**：

| 任务复杂度 | 预估时间 |
|:---|:---|
| 简单（复制粘贴） | 30分钟 |
| 中等（需要思考） | 1-2小时 |
| 复杂（需要设计） | 4小时 |
| 非常复杂（需要研究） | 1天 |

**缓冲时间**：
- 总时间 = 任务时间累加 × 1.2（20%缓冲）

### Step 5: 生成任务清单

**格式**：

```markdown
### Epic 1: [功能模块名]

| ID | 任务 | 依赖 | 优先级 | 状态 | 预估 | 负责人 |
|----|------|------|--------|------|------|--------|
| T1.1 | 任务描述 | - | P0 | ⏳ | 30m | |
| T1.2 | 任务描述 | T1.1 | P0 | ⏳ | 1h | |
```

---

## 任务粒度标准

| 粒度 | 时间范围 | 示例 | 验收标准 |
|:---|:---|:---|:---|
| **Epic** | 1-2周 | "用户登录模块" | 整个模块可用 |
| **Task** | 4-8小时 | "实现微信登录" | 功能可用 |
| **Subtask** | 30分钟-2小时 | "调用微信API获取code" | 代码可运行 |

---

拆解完成后，检查：

- [ ] 每个任务都有明确的验收标准
- [ ] 任务粒度合适（30分钟~4小时）
- [ ] 依赖关系清晰
- [ ] 优先级设置合理
- [ ] 时间估算合理
- [ ] 没有遗漏关键任务

## 核心原则

### 1. 自动开始
- 拆解完计划后，**立即自动开始执行第一个任务**
- 不需要等待用户确认
- 不需要用户说"开始执行"

### 2. 逐步推进
- 完成一个任务后，**自动执行下一个可执行任务**
- 按照依赖关系和优先级顺序执行
- 持续执行直到所有任务完成

### 3. 上下文理解
- 每次执行前，理解对话上下文
- 结合之前的拆解结果
- 考虑前后任务的关联

### 4. 持续完成
- 持续执行直到完成用户描述的目标
- 除非用户明确停止，否则不中断
- 遇到阻塞自动寻找解决方案

   ↓

   - 调用相应工具/能力

   - 标记状态：🔄 → ✅
   - 更新执行日志

   - 优先级是否足够？
   ↓

   - 重复步骤3-5
   ↓

   - 生成完成报告

## 任务执行规则

### 可执行任务判断

一个任务可以执行，当且仅当：

1. **状态为"⏳待开始"**
2. **所有依赖任务已完成**（依赖列表为空或所有依赖任务状态为✅）
3. **优先级为P0或P1**（P2任务可以延后）
4. **没有阻塞**（没有被标记为🚫阻塞中）

### 任务执行顺序

1. **优先级优先**：P0 > P1 > P2
2. **依赖优先**：先执行依赖少的任务
3. **Epic顺序**：按Epic顺序执行
