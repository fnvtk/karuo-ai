# Soul 派对成片工作流：从零下载到 SEO 片尾

> **定位**：**先转录与分段清单 → 再切片 → 再增强**；增强侧拆成「洁净 → 叙事 → 呈现」三块，并在块内按**可验收子步骤**写清（封面、字幕、人声、故事、节奏、重点、尾帧 SEO）。  
> **高光提取**：**禁止 Ollama**；仅用 **OPENAI 兼容 API 队列**（`OPENAI_API_BASES` / `OPENAI_API_KEYS` / `OPENAI_MODELS`）选当前最佳模型，失败则规则切分；或由 **Cursor 直接制作** `highlights.json`（见 §4 与 `identify_highlights.py` 文件头）。  
> 规则真源：`Soul竖屏切片_SKILL.md`、`参考资料/`、脚本 `--help`。每一步的执行都要告诉我执行到哪一步？下一步做什么。百分比以及还要多少的时间  
> **路径**：本文所在目录 = `木叶_视频内容/视频切片/`（「本 Skill 根目录」）。  
> **目录索引**：同目录 [`README.md`](./README.md)。  
> **剪映三类对照（字幕 / 封面 / 特效）**：[`Soul成片三件套_剪映对照_总览_SKILL.md`](./Soul成片三件套_剪映对照_总览_SKILL.md)（**M01k**）+ 子 Skill **M01l～M01n**，保证流程分类清晰、单场验收不丢项。  
> **追问自检**：[`参考资料/成片流程_追问与自检.md`](./参考资料/成片流程_追问与自检.md)；执行节奏上若需报进度，在答完当日阶段追问后再报「百分比/剩余时间」更可验。

## 〇、总流程（逻辑顺序，禁止跳步）

```
下载原片 → 塑形标定(analyze)
    → 【阶段Ⅰ】提取音频 + 转录稿洁净
            → 用最佳模型做「热点分段清单」（时间节点 + 文案要素齐全，见附录 A）
            → 校对后落地为 highlights.json（未通过不得切片）
    → 【阶段Ⅱ】仅 batch_clip →「切片/」横版素材
    → 【阶段Ⅲ】增强（分三部分、多子步）→「成片/」
    → 【阶段Ⅳ】SEO 两页尾帧（`soul_slice_pipeline --two-folders` **默认自动**；仅跑 `soul_enhance` 时须手跑 §七 或重跑流水线成片段）
```

**硬性约定**：

1. **必须先有合格 `切片/`**，再跑 `soul_enhance`；不得在「未切片」状态下跳过叙事边界去赌成片。  
   **竖条宽度**：**阶段Ⅲ 之前**须完成 **`裁剪检查/` 塑形标定**（见 §三），把 **CROP_VF / OVERLAY_X** 写入 `soul_enhance`；缺此步易导致竖条裁错、人脸或飞书侧栏被切没。
2. **增强内部的顺序**：先保证**口播与字幕源干净、叙事成条**，再谈封面大字与逐字特效；**尾帧 SEO 永远最后**，且尾帧**无声**；成片正片结尾须保留**可听人声/信息收口**（避免「画面结束了声音还空一大段」或「声音突然断在空白里」）。**片尾四层**（人声窗 → 有声 `cta_ending` → 无声 SEO×2 → 状态防双尾）见 **`Soul成片片尾_SKILL.md`（M01o）**；`soul_enhance` **不**内嵌 SEO 拼图，SEO 仅 `append_seo_keyword_tail` / 流水线。
3. **违规词**：平台规则与广告法敏感项以**人工终审**为准；脚本侧只做通用清洗（见阶段Ⅰ），**高危词须在 `highlights.json` 与口播字幕定稿前删掉或改写**。

## 三、裁剪检查（塑形标定 / CROP_VF）— **必经一环**

> **口语备忘**：有人把本步误说成「财经警察」等——**指的就是这一步**：用 **`analyze_feishu_ui_crop.py`** 在录屏上找 **深色主内容区包络**，定 **竖条裁剪宽度** 与 **横版上的叠加 X**，**不是**另跑财经类脚本。与万推 **`Soul剪辑取向分析`** 为**同源几何**（先判竖条/全幅策略时也可对照该 Skill）。

**本环节不得省略**：一体化 `soul_slice_pipeline` **不会**自动跑本节；**阶段Ⅲ `soul_enhance` 前**务必先做，否则默认 `crop` 可能与当场飞书布局不符。

**产物（与第 98 场等历史目录一致，便于肉眼验收）**，均在 `场次_output/裁剪检查/`：

| 文件 | 作用 |
|------|------|
| `*_全画面_20%.png` | 取样帧全画面（文件名里 `20%` 对应 `--at 0.2`） |
| `*_竖条塑形预览_Wx1080.png` | 竖条效果预览，**W** 为成片竖条像素宽（如 522） |
| `*_塑形裁剪参数.txt` | **CROP_VF**、**OVERLAY_X**、**OUTPUT_SIZE**，复制进 `soul_enhance` |

```bash
VIDEO="$HOME/Movies/soul视频/原视频/第137场_20260330.mp4"
OUT="$HOME/Movies/soul视频/第137场_20260330_output"
SCRIPT="/Users/karuo/Documents/个人/卡若AI/03_卡木（木）/木叶_视频内容/视频切片/脚本"

mkdir -p "$OUT/裁剪检查"
python3 "$SCRIPT/analyze_feishu_ui_crop.py" "$VIDEO" --at 0.2 --save-dir "$OUT/裁剪检查"
# 竖条内仍见浅灰/白边时，在同样命令上加左右内收（像素，可 12～36 试到满意）：
# python3 "$SCRIPT/analyze_feishu_ui_crop.py" "$VIDEO" --at 0.2 --edge-shrink 10 --edge-expand 32 --save-dir "$OUT/裁剪检查"
# 竖条太窄、两侧头像被吃：减小 edge-shrink 或增大 edge-expand；太宽泛白则略增 edge-shrink
```

1. 打开 **`竖条塑形预览`** 与 **`全画面`** 对照：主画面、侧栏、头像是否裁得合理。  
2. 终端与 **`塑形裁剪参数.txt`** 中的 **CROP_VF**、**OVERLAY_X** 原样用于阶段Ⅲ（示例见 §六命令）。  
3. **新场次禁止照抄旧场数字**；布局变了就重新跑本节覆盖 `裁剪检查/`。

### 文件名与封面大字：**仅 2 字或 4 字 · 通俗名词 · 不重复**（强制口径）

> 与 `Soul竖屏切片_SKILL`、脚本 `batch_clip.py` / `soul_enhance.py` 一致：**封面与 `切片/`、`成片/` 的 mp4 主名**由 **`cover_text_for_filename()`** 生成，与封面大字**同源**：优先 **`viral_hook` → `hook_3sec` → `title`**（再回退高光内其它字段/切片名）。**不是**「`file_stem` 单独决定文件名」；`file_stem` 仍建议填写，与 hook/title **字面互异**，便于元数据与附录对照。

| 字段 | 用途 | 字数与风格 | 不重复 |
|------|------|------------|--------|
| **`file_stem`** | **元数据/附录**用的短主题名；与 `viral_hook` 等**同规则**书写（成片文件名**不**单独以它为准，见上注） | **仅 2 个汉字或 4 个汉字**，通俗名词 | **整场各条互异**；与同条 `viral_hook`、`hook_3sec`、`title` **字面互异** |
| **`viral_hook`** | 封面首屏大字（**第一优先**） | 同上 **仅 2 字或 4 字**，**通俗名词**，可略偏「吸睛」但仍要长辈/外行能懂 | **整场互异**；与同条 `file_stem`、`hook_3sec`、`title` **字面互异** |
| **`hook_3sec`** | 前 3 秒口播/备选封面字（**第二优先**） | 同上 **仅 2 字或 4 字**，**通俗名词** | **整场互异**；与同条其余三字段 **字面互异** |
| **`title`** | 列表/元数据/封面**第三优先** | 同上 **仅 2 字或 4 字**；长说明放 `question` / `transcript_excerpt` / 草稿 md | **整场互异**；与同条 `file_stem`、`viral_hook`、`hook_3sec` **字面互异** |

**竖屏封面字号**：`soul_enhance.py` 内按**可见字数**计算起始字号（约 **字少更大、字多更小**，28～100px），仍会在左右 padding 内自动换行/略降字号防溢出；细则见 `Soul成片封面_SKILL.md`。

**怎么写**：优先 **2 字名词**（好记、好念）；同场需区分度时可部分条用 **4 字名词**（如「私域运营」「飞书会议」）。忌「某某说三句话教你 XXX」长句。定稿时用**表格自检**：`序号 | file_stem | viral_hook | hook_3sec | title` —— **无重复字面、每条四字仅 2 或 4 汉字、无生僻字**。

**与附录 A 的关系**：附录 A 清单可长；**落地 JSON 时**四字段必须压成 **2 或 4 字名词** 并满足本表，再进阶段Ⅱ。

**塑形竖条与头像**：若竖条**偏窄**、**两侧头像或参会条被切掉**，不要硬抄旧场 `CROP_VF`：① **减小** `--edge-shrink`（内收越少画面越宽）；② 或加 **`--edge-expand N`**（左右各外扩 N 像素，脚本已支持）。改完必须打开 **`裁剪检查/*_竖条塑形预览_*.png`** 与 **全画面对照**，确认左右人物/头像可见后再把新 **`CROP_VF` / `OVERLAY_X`** 写入 `soul_enhance` / 流水线。

### Cursor 内「文案抽象层」：**少追问 · 直接出词 · 可检索 · 接地气还显高级**

> **默认优先在 Cursor 完成**：标题/封面/钩子四字段的**立意与用字**在 **Cursor 对话**里一次性定稿（对照 `transcript.srt` 段意 + 本条叙事），**尽量不要再拆成多轮问卷**；由助手直接给出可粘贴进 `highlights.json` 的成品词表（或整文件 JSON）。**不依赖** `identify_highlights` 的 API/规则去「自动生成」这四字，除非本场次硬性规定必须跑脚本出初稿后再由 Cursor 精修。

1. **抽象层**：每条先想清楚「这一条到底在讲什么**一个**核心命题」，再落到四个**不同字面**的 **2 字或 4 字通俗名词**（见上表）；`transcript_excerpt` / `reason` 可略长，**封面与文件名只用四短字段**。  
2. **吸引力（可对照互联网）**：拟词前可做**简短检索**（如「短视频钩子词」「爆款标题 动词」「私域 短视频 开头」），把常见结构**内化**为短词：  
   - **情绪 + 动作**（真、翻、响、醒、别装睡…）  
   - **价值承诺**（省到肉、给你声、吃长尾…）  
   - **认知冲突**（反着学、抄不走、别白割…）  
   - **稀缺/获得感**（独一味、卡位早、势在那…）  
   落地时须 **Soul/私域场域可播、过审优先**：避免过度低俗、辱骂、绝对化功效与违规金融承诺；「狠词」用**大白话里的张力**，不碰平台红线。  
3. **质感口径**：在**口语、接地气**与**略高级、像内行黑话**之间取平衡——读起来像真人聊天，又像懂行的提炼；忌堆砌网络烂梗与同义反复。  
4. **与脚本分工**：`identify_highlights.py` 仍可用来**粗分段时间轴**（尤其整场很长时）；**四短字段**以 Cursor 定稿为准，脚本跑出长标题后**必须**按上表压成 **2 或 4 字通俗名词**并重检去重。  
5. **重剪保留文案**：若已有一份满意的 `highlights.json`，整场重切/重增强时应用 `soul_slice_pipeline.py **--skip-transcribe --skip-highlights**`（或分步只跑 `batch_clip` + `soul_enhance`），**勿**加 `--force-transcribe` 以免覆盖精修 JSON；除非刻意要从新转录整条时间轴再起稿。

## 四、阶段Ⅰ：转录 + 整场稿 + **热点分段清单**（**先于切片**）

**目标**：① 从视频得到**带时间轴、且通顺可读、语义前后一致**的整稿（含标点分段与通篇纠错）；② 用**最佳可用模型**把整场拆成 **6～8 个**热点段，**每一条都写明起止时间、标题、要点与信号**，并**去重**；③ **校对通过并写入 `highlights.json` 后**，才允许进入阶段Ⅱ `batch_clip`。

| ------- | ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Ⅰ-1** | **从原片提取音频** + **MLX 转录**                 | `ffmpeg` 提音频（流水线内）；`mlx_whisper` 出 `transcript.srt`。可单独执行：先跑流水线至仅生成 SRT（见下方「仅转录」）                                                                                                                                                                                                              |
| **Ⅰ-2** | 繁转简、常见 ASR 错词替换、**语助词弱化/删除**             | `soul_slice_pipeline.py` 内 `transcript_to_simplified`（对齐 `Soul竖屏切片_SKILL`）                                                                                                                                                                                                                     |
| **Ⅰ-3** | **违规与敏感表达**（绝对化、医金违规承诺、辱骂歧视等）            | **人工**通读或抽查 `transcript.srt`；高危段在后续时段中避开或改文案                                                                                                                                                                                                                                                   |
| **Ⅰ-4** | 整场稿 **标点分段 + 通顺通俗 + 上下文一致 + 通篇错别字/用词纠错** | **Ⅰ-2 为规则层**（繁简、常见错词、语助词）；**Ⅰ-4 为整稿语言层**：通读 `transcript.srt`，理顺语句、统一术语与指代、消灭错别字与别扭用词，且**前后文意思一致、不断章改义**。操作细则见 **§4.2.1**                                                                                                                                                                       |
| **Ⅰ-5** | **热点分段 + 全要素文案**（**必须先于切片**）             | 用**附录 A** 提示词 + **带时间戳的发言稿**（可用 `identify_highlights.py` 内部同款：`srt_to_timestamped_text` 逻辑，或直接贴 SRT）调用**当前最佳模型**，输出 **6～8 条**、**按时间排序**、**同话题只保留最精华 1 段**。每条须含：**时间戳 `HH:MM - HH:MM`（可与 SRT 对齐后细化为 `HH:MM:SS`）**、**时长**、**标签 1～2**、**标题 ≤15 字（网感，仅草稿）**、**内容 3～5 条 bullet（每条 ≤25 字）**、**信号解读 1 句** |
| **Ⅰ-6** | **清单 ↔ SRT 对表**                          | 人工核对每条起止是否落在**讨论最激烈、信息密度最高**处（多人追问、补充、数字密集）；修正边界                                                                                                                                                                                                                                               |
| **Ⅰ-7** | **落地为 `highlights.json`**                | 将附录 A 每条映射为 JSON 的 `start_time` / `end_time`（与 `batch_clip` 一致，常用 `HH:MM:SS`）、`title`、`hook_3sec`、`viral_hook`、`file_stem`、`question`、`cta_ending` 等（字段约定见 `Soul竖屏切片_SKILL` + `流程说明_高光成片.md`）。**必须执行 §三 下「文件名与封面大字」表**：`file_stem` / `viral_hook` / `hook_3sec` / `title` 均为 **仅 2 字或 4 字通俗名词、整场+同条内字面不重复**。**定稿前不得运行 `batch_clip`**                                                                                                  |


### 4.2.1 Ⅰ-4 详解：通顺、上下文、错别字（整稿语言层）

在 **Ⅰ-2 机器清洗之后**、**Ⅰ-5 热点分段之前**，对 `transcript.srt` 做一次**面向整场的语言整理**（人工为主，或可辅助大模型，但必须守住时间轴）。

1. **标点与分段（结构）**
  - 补句号、问号、逗号停顿，拆「一屏过长」的句子，便于模型按时间戳对齐话题。  
  - **合并两条字幕或改动起止时间**时务必谨慎：改完用播放器或 `ffplay` 抽查，避免**字幕与声音错位**。
2. **通顺与通俗（可读）**
  - 按**口语听感**理顺表达：删重复赘词、补全省略但不**编造事实**。  
  - 目标：**单条字幕念起来顺**、整场连起来**像同一场对话**，而不是 ASR 碎块堆砌。
3. **上下文一致（语义）**
  - **指代**：「这个/那个/他/他们」结合前后句还原为可理解的主语或宾语（必要时轻量改写，**不改变原说话人立场与事实**）。  
  - **术语与数字**：同一概念全场用词统一（如产品名、平台名、金额单位）；数字与「万/亿/元」等与口播一致。  
  - **禁止**：为通顺而**歪曲原意**、把未说过的结论写进稿里。
4. **错别字与用词（通篇纠错）**
  - 通篇扫：**错别字、近音误识、形近字**；专名、英文产品名的固定写法。  
  - 与 **Ⅰ-2** 的 `CORRECTIONS` 规则表互补：规则表没有的，在 **Ⅰ-4** 收口。
5. **与大模型协作时的红线**
  - **优先只改「字幕文本行」**，保留每条 `00:00:00,000 --> 00:00:00,000` 时间码不变，避免烧录/对齐全线漂移。  
  - 若必须整文件重排轴，应用可信 SRT 工具或脚本**校验时间单调、无重叠**后再进入 Ⅰ-5。

## 六、阶段Ⅲ：增强（三部分 + 八步细节）

**前置**：已完成 **§三 `裁剪检查/`**；`--crop-vf` / `--overlay-x` **必须从** `裁剪检查/*_塑形裁剪参数.txt` **粘贴本场数值**，勿手填旧场示例。

**入口命令**（成片输出到 `成片/`；下例中 `crop`/`overlay` 请换成你 txt 里那一行）：

```bash

OUT="$HOME/Movies/soul视频/第137场_20260330_output"

  -c "$OUT/切片" \
  -l "$OUT/highlights.json" \
  -t "$OUT/transcript.srt" \
  -o "$OUT/成片" \
  --vertical --title-only --force-burn-subs --typewriter-subs \
  --crop-vf "crop=752:1080:416:0" --overlay-x 416 \
  --silence-gentle
```

- **逐字跟读**：务必带 **`--typewriter-subs`**。有词级时间轴（Whisper 词级 SRT）时，脚本在**每个词的时间段内**再按字分配显现节奏；**金黄关键词**（`soul_enhance.py` 内 `KEYWORDS` 列表，与烧录高亮同源）在逐字时间轴上**自动略增单字权重**，重点字多占一瞬，视觉上更「钉」住口播，跟读更丝滑。  
- **听感丝滑 / 语速**：成片默认音画同倍率 **×1.10** 加速（`atempo`+`setpts`）。要更顺、少变调可 **`--speed-factor 1.04`** 或 **`1.06`**；尾帧拼接或极介意音色时用 **`--no-speedup`** 原速。

（可选：在上一命令末行并列加 `--keyword-pins` → **弱版** 1～2 条浮层关键词，**不等于** §6.3 **第 6′ 步** 的 3～5 点渐进干货。）

以下「三部分」是**工作流上的验收分层**；检查单含 **第 5～9 步** 与 **第 6′ 步**（干货渐进条）。底层由 `soul_enhance.py` 聚合实现；**第 6′ 全量**见 **§6.3.1** 与后续脚本迭代。

- **信息**：`cta_ending` 丝滑，忌硬广口令堆砌。

### 6.3.1 干货渐进条：目标规范 · 数据从哪来 · 与脚本现状

**目标（验收口径）**

1. **条数**：每条成片 **3～5 条**干货（少于 3 视为信息密度不足、除非本段本就极短）。
2. **形态**：每条显示为 **序号 + 短语**（如 `1. 二次验证是卡点`），可带统一前缀区「要点」「记住」等，**忌长段落**。
3. **时序**：第 `k` 条在**口播进入对应论点时**再弹出；允许**前一条仍留在屏上**（堆叠或列表加长），或**短显后保留为淡色列表**——以「**跟随整条叙述节奏**」为准，禁止开场 3 秒一次性甩满 5 条。
4. **与主字幕关系**：主轨仍是**逐字跟读**；干货条是**二次提炼**，内容须能在当前前后文里**找到依据**，不编造。

**建议在 `highlights.json` 单条 clip 中预留（待脚本消费）**

示例（字段名可后续与 `soul_enhance` 对齐）：

```json
"takeaway_pins": [
  { "n": 1, "text": "二次验证是卡点", "t_start_sec": 18.5, "t_end_sec": 45.0 },
  { "n": 2, "text": "先跑通再放量", "t_start_sec": 46.0, "t_end_sec": null }
]
```

- `t_start_sec` / `t_end_sec`：相对**本切片文件开头**的秒数（与 `soul_enhance` 裁静音后的时间轴一致时，需在同一基准上校准）。  
- `t_end_sec` 可空表示「显示到下一条出现前」或「固定露 2～3 秒」由渲染策略定。

**与当前 `soul_enhance.py` 的关系（避免误以为已全自动）**

- 现有 `**--keyword-pins`**：调用 `apply_keyword_pin_overlays`，从 `hook_3sec` / `title` / `transcript_excerpt` 取 最多 2 条短句，在**启发式时间点**各显示约 **2.8s**，**不是** 3～5 条序号、**也不是**按你口述进度逐条绑定。  
- **要完全达到第 6′ 步**，需要在 `soul_enhance`（或前置步骤）中：**读取 `takeaway_pins` 类似结构 → 按 `t_start_sec` 叠 PNG/ASS**。属**待开发/待迭代**；在此之前可用 **剪映/PR 按时间码手叠** 或 **先用 `--keyword-pins` 做弱版信息条**。

**成片命令可带弱版关键词条（可选）**

```bash
python3 "$SCRIPT/soul_enhance.py" \
  … \
  --keyword-pins
```

（与 `--vertical` 等并列；完整参数以 `soul_enhance.py --help` 为准。）

## 七、阶段Ⅳ：片尾总结构 + SEO 两页（最后一步）

**专类 Skill**：`Soul成片片尾_SKILL.md`（**M01o**）。本节是命令级摘要。

### 7.1 为什么有时「成片里没有 SEO 图」

- **`cta_ending` 字幕条**在 **`soul_enhance` 内**烧录，与 **SEO 静帧**无关。  
- **两张 SEO 页**只由 **`append_seo_keyword_tail.py`** 拼到文件末尾；**仅跑 `soul_enhance` 不会触发**。  
- **`soul_slice_pipeline --two-folders`** 在增强成功且存在 **`参考资料/视频尾帧_SEO关键词200.txt`** 时**默认**调用 append；可用 **`--no-seo-tail`** 关闭，用 **`--seo-tail-force`** 传 **`--ignore-state`**（须已换回**无尾帧母片**）。

### 7.2 片尾时间轴（观众侧）

1. **人声与节奏**：最后一句讲完；片尾保留约 **2.85s** 静音窗不剪死（见 §6 第 7 步）。  
2. **有声 CTA**：`highlights.json` 的 **`cta_ending`**，在口播收尾段展示；脚本内 **`CTA_HOLD_AFTER_VOICE_SEC`** 让人读完再微延到无声。  
3. **无声 SEO ×2**：默认 **`--pages 2`**，总时长约 **`--duration`**（如 2.8s 则每页约 1.4s）；第二页底默认 **`点头像进房 · 每晚派对直播`**（`--no-action-line` 可关）。  
4. **防双尾**：`成片/.soul_seo_tail_state.json` 记录时长；再次运行且时长一致则**跳过**，避免叠两段 SEO。

### 7.3 命令（定稿后、已备份无尾帧母片）

```bash
python3 "$SCRIPT/append_seo_keyword_tail.py" \
  --dir "$OUT/成片" \
  --keywords "$SCRIPT/../参考资料/视频尾帧_SEO关键词200.txt" \
  --duration 2.8 \
  --pages 2 \
  --per-page 8
```

| 验收项       | 说明                                                                            |
| --------- | ----------------------------------------------------------------------------- |
| **隐藏化**   | 双页 RGBA 高透明 + 字色贴近底色，**观感上尽量「看不见」**，仍利于搜索场景长尾词覆盖                              |
| **无声**    | 尾帧拼接段**无音轨增益**；正片最后一句应在**进入尾帧之前**讲完；误用 **`--ignore-state`** 于仍含尾帧的文件会 **叠双尾** |
| **两页都进片** | **`--pages 2`** 时 ffprobe 总时长应比无 SEO 母片长约 **`--duration`**；肉眼末尾有两段极淡静帧切换              |
| **封面底模糊** | 正片封面约 **10% 轻模糊**由 `soul_enhance` 控制；与 SEO「全透明藏字」是**两套逻辑**                     |
| **勿重复跑**  | 改片须用**无 SEO 备份**为输入；依赖状态文件或先恢复母片再跑                                        |

可调：`append_seo_keyword_tail.py` 顶部 `_WASH_ALPHA` / `_TITLE_ALPHA` / `_BODY_ALPHA`、命令行 `--duration`、`--per-page`、`--action-line`。

## 附录 A：阶段Ⅰ-5 热点分段提取（提示词 · 先清单后切片）

将下方整段作为 **System / 用户提示词** 的一部分，**用户消息**附上「带时间戳的整场转录」（推荐直接贴 `transcript.srt` 或已由脚本拼好的 `HH:MM:SS 文本` 逐行稿）。

```markdown
# 角色
你是资深会议内容策划与短视频选段编辑，擅长从长录音转写稿中，按时间戳找出**最适合做热点切片**的片段，并输出**可执行的细分段清单**（尚未 ffmpeg 切片）。

# 任务
阅读附带「带时间戳的发言/转写文本」，完成：
1. **时间节点筛选**：识别讨论最激烈、信息密度最高的片段（优先：多人互动、追问、补充、数字密集处）。
2. 输出 **6～8 个**热点段，**按时间顺序排列**；**同一话题只保留最精华 1 段**（去重，不重复相似内容）。

# 内容筛选标准（命中越多越好）
- 有具体数字的商业案例（如「几毛成本卖 30 元」「1200 万撬 1.2 亿」）
- 有明确方法论的分享（如「三步走」「四要素」）
- 有冲突/反转/意外的故事（如「14 岁做号商 17 岁买玛莎」）
- 有可复制的操作路径（如「500 台手机月入 XX」）
- 有风险警示的内容（如「二次验证是卡点」）

# 每条输出格式（严格遵守）
- **时间戳**：`HH:MM - HH:MM`（起止；若稿为 HH:MM:SS 可细化到秒）
- **时长**：`X分钟`（或 `X分Y秒`）
- **标签**：1～2 个核心标签（如「暴利链路」「风险预警」「思维模型」）
- **标题**：清单草稿可不超过 15 字、网络热词风格（如「搞钱主线」「降维打击」）；**写入 `highlights.json` 时**须再压成 **2 或 4 字通俗名词**，并与 `file_stem` / `viral_hook` / `hook_3sec` **同条异字、全场不撞**（见正文 §三）
- **内容**：3～5 条 bullet，每条不超过 25 字
- **信号解读**：1 句话总结该片段的核心洞察价值

# 数量与顺序
- **6～8 条**，按时间从早到晚。
- **去重**：同一话题只保留最精华 1 个片段。

# 输出语言
全文 **简体中文**。
```

**与 `highlights.json` 的衔接**：将每条的时间戳换算为 `batch_clip` 使用的 `start_time` / `end_time`（通常为 `HH:MM:SS`）。**短名四件套**（均 **2 或 4 字、通俗名词、不重复**）：`file_stem`（文件名）、`viral_hook`（封面首优）、`hook_3sec`、`title`（封面第三优先）——见正文 §三 自检表。bullet 与信号解读可写入 `transcript_excerpt` / `reason` 或单独 `热点分段清单_草稿.md` 备查。
