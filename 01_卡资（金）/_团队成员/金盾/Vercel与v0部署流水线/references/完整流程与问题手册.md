## 二、Git 主从与 v0 连接（默认链路）

- **主来源**：**本地上帝之眼项目 → 推送到 GitHub fnvtk/godeye** 的 **main** 分支为唯一主源；v0 上连接的 Git 即该仓库。
- **默认链路**：**从本地/GitHub 到 v0**，不要以 v0 为源头「往下降」。即：本地改 → push main → (可选) 在 v0 中基于 main 继续；**不要**默认从 v0 单方面往 GitHub 推为主。
- **若无 v0 分支或未同步**：以**本地 GitHub（main）**为准；本地 main 即当前项目主体。
- **v0 上该项目的当前状态**：应以**本地已上传的上帝之眼 GitHub（main）** 覆盖/修改为当前项目主体。即在 v0 的 Git 里：
  - 连接仓库：`fnvtk/godeye`（与本地一致）；
  - 以 **main** 为基准：在 v0 中若有「从 main 拉取」「Reset to main」「Use main as base」等，用 main 覆盖当前 v0 对话的代码状态，使 v0 上的「上帝之眼」与本地 push 上去的 main 一致。
- **小结**：v0 创建的 Git 分支（如 `v0/fnvtk-xxx`）用于在 v0 里编辑时合并回 main；**主源始终是 main（本地项目）**，v0 项目界面/代码应以 main 为主、用 main 覆盖成当前项目。

### 三段式同步（本地 = GitHub main = v0，不用现有 v0 状态）

要保证 **v0 是本地的且最新的前端**，且 **v0 配置的 Git 页面也是最新的前端**（不用现有的 v0 对话状态），按以下顺序做**一次完整协同**：

| 段 | 操作 | 说明 |
|----|------|------|
| ① 本地 → main | 在项目根执行 `node scripts/push-local-frontend-to-main.js "说明"` | 将本地 frontend 等变更 commit 并 push 到 fnvtk/godeye 的 main，使 **main = 本地最新前端** |
| ② main 已最新 | 上一步完成后 GitHub main 即本地最新 | 无需额外操作 |
| ③ v0 = main | 在 v0 该上帝之眼对话 → 左侧 **Git** → 点击 **「Pull from main」/「Sync from main」/「Reset to main」**（以 v0 界面为准） | 使 **v0 当前对话与 Git 页面均为 main**，即本地最新前端；**不要用现有 v0 状态**，用 main 覆盖 |

完成后：**本地 = GitHub main = v0**，三段斜动协同一致。

- **脚本**：`上帝之眼/scripts/push-local-frontend-to-main.js`（只做 ①；③ 需在 v0 网页操作）。

## 六、常见问题与解决

| 现象 | 原因 | 处理 |
|------|------|------|
| 未设置 VERCEL_TOKEN | .env 缺失或未读 | 从 账号与API索引 § Vercel 复制到 上帝之眼/.env |
| CREATE project failed / Git | Vercel 未连 GitHub | 在 Vercel 后台用 GitHub 登录并连接仓库（一次性） |
| 预览/生产跳转登录 | 部署保护开启 | 脚本已自动 PATCH 为仅预览需登录；若仍跳转，检查 ssoProtection 为 preview |
| 未设置 V0_API_KEY | .env 无或 v0 未读 | 从 账号与API索引 § v0.dev Secret 填入 .env，或使用 https://v0.app/chat/settings/keys 的 Platform API Key |
| v0 401 | V0_API_KEY 无效或非 Platform | 使用 v0 设置中的 API Key，或确认 Secret 可用于 Platform API |
| v0 项目已存在 | 之前创建过 | 脚本会识别并输出 webUrl，无需重复创建 |
| 线上打不开 | 构建未完成或保护 | 稍等再访问；确认生产已公开（见上） |
| 优化对话被重复创建 | 未先检查 | 使用 create-v0-optimization-chat.js：先 GET 项目 chats，有「前端优化计划」或最近一条则复用，不新建 |
| 对话名是英文 | 未传 name | 创建时传 `name: '前端优化计划'`（脚本已改为中文名） |

## 七、提醒清单（执行前必看）

1. **先 Vercel 后 v0**：必须先部署成功且生产可访问，再执行 sync-to-v0。
2. **v0 项目正确优先**：先确认 v0 项目页、名称、vercelProjectId 正确，再做其它优化。
3. **全链路无网页**：部署与 v0 关联均通过 API + Token 完成，不打开浏览器。
4. **Token 统一从金盾/索引取**：需要时从 账号与API索引 取 VERCEL_TOKEN、V0_API_KEY，或确认上帝之眼/.env 已同步。
5. **本地前端位置固定**：v0 优化后代码必须落回 **项目根/frontend**，不要搞错路径；再用 sync-after-v0.js 同步。
6. **做完任务要反馈到项目**：执行 sync-after-v0 后结果会自动写入「同步与反馈记录」；其它部署/整理操作结果反馈给卡若。
7. **Git 以本地/GitHub main 为主**：v0 连接的 Git 即 fnvtk/godeye；默认链路从本地 main 到 v0，不以 v0 往下降为主；v0 上该上帝之眼项目应以 main（本地已上传）覆盖为当前项目主体。
8. **三段式同步**：要 v0 与 Git 页都是本地最新前端（不用现有 v0 状态）时，先运行 `push-local-frontend-to-main.js` 使 main=本地，再在 v0 Git 面板「从 main 拉取」使 v0=main。
