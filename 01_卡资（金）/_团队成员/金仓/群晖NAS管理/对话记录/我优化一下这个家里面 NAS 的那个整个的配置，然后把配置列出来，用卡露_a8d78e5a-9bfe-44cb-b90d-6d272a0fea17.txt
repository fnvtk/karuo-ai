## 四、性能问题与降负载方案（把负载降下来）

目标：**把 CPU 与整体负载降下来，当全新 DiskStation 用**。

### 4.1 第一步：找出谁在占 CPU

- **DSM**：资源监控 → 进程，按 CPU 排序，记下进程名或对应套件。  
- **SSH**：`top`，按 P 按 CPU 排序；或 `ps aux --sort=-%cpu | head -20`。

### 4.2 常见高负载来源与处理

| 来源 | 处理办法 |
|------|----------|
| **索引/媒体扫描** | 控制面板 → 索引服务：只索引必要文件夹；Photo/Video 只选必要相册；或临时关闭索引，等空闲再开。 |
| **备份/同步任务** | 控制面板 → 计划任务：把 Hyper Backup、Cloud Sync、Synology Drive 等放到凌晨或空闲时段，避免多任务同时跑。 |
| **Docker 容器** | Container Manager：对占 CPU 的容器设「资源限制」上限（如 CPU 50%、内存 512MB）；不用则停止或删除。 |
| **杀毒/安全扫描** | 安全顾问：扫描改到低峰；不需要实时保护可关或缩小范围。 |
| **下载/BT** | Download Station 等：限速、限并发；或低峰再下。 |
| **大量小文件读写** | 避免同一时间全盘扫描、全量同步；分批或错峰。 |

### 4.3 可直接执行的降负载步骤（按顺序做）

1. **计划任务**：控制面板 → 计划任务 → 禁用或删除不必要任务；保留的改为凌晨/空闲执行。  
2. **索引**：控制面板 → 索引服务 → 取消不需要的共享文件夹索引；Photo/Video 缩小索引范围。  
3. **套件**：套件中心 → 停用或卸载不用的套件（如不用的 Download Station、Video Station 转码等）。  
4. **Docker**：Container Manager → 对每个容器设 CPU/内存上限；不用的容器停止。  
5. **安全顾问**：安排扫描在低峰；可关闭实时保护或缩小范围。  
6. **重启**：DSM 与套件更新到最新后，重启 NAS 一次，清异常进程与缓存。  

### 4.4 效果验证

- 优化前后在「资源监控」看一段时间（如 10～30 分钟）的 CPU 与 1/5/15 分钟负载。  
- 若仍高：用 `top` 或资源监控记下进程名 → 对应到套件或容器 → 再按上表限制或停用。

**和 DS1825+ 的真实性能对比（直接说结论）**

你当前这台 192.168.110.29 没在文档里写具体型号，只能按「常见家用 DiskStation」来估。DS1825+ 是 2025 年的 8 盘位企业/进阶机型，所以和 .29 的真实差距大致是这样：

**内存**：DS1825+ 标配 8GB DDR4 ECC，可扩到 32GB。老款 2～4 盘位很多是 2GB 或 4GB 焊死，开多套件、Docker、Photo/Video 索引时更容易吃满、换页，从而拖慢整机。

**盘位与扩展**：DS1825+ 是 8 盘位 + 可再扩，还有 M.2 SSD 缓存和 2.5G 网口，顺序读写能到 1.5～2.2 GB/s 级别。.29 若是 2～4 盘位、无 SSD 缓存、千兆网，实际顺序读写和随机 IO 都会弱一截，大量小文件或同时多任务时体感更明显。

**网络**：DS1825+ 双 2.5GbE，内网大文件拷贝和多人同时用更够用。.29 若是单千兆，在备份、同步、多端同时访问时更容易成为瓶颈。
