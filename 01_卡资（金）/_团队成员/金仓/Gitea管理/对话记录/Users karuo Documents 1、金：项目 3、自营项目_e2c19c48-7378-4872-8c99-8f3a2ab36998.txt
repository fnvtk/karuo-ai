# 01_扫描模块

> 核心任务：快速、精准地发现目标网段中的可用IP和开放服务

---

## 一、模块概述

扫描模块负责分布式算力矩阵的「节点发现」环节，通过多种扫描技术组合，实现从IP发现到资产画像的完整流程。

## 二、四大扫描板块

### 板块1：IP段扫描（广度扫描）

**目标**：快速扫描大范围IP段，发现存活主机

| 项目 | 说明 |
|:---|:---|
| 工具 | masscan（速度优先）/ zmap（大规模） |
| 输入 | IP段列表（CIDR格式，如 192.168.1.0/24） |
| 输出 | 存活IP列表 + 开放端口 |
| 速率 | 支持自定义速率（默认 10000 pps） |

```bash
# 示例：扫描C段
masscan 192.168.1.0/24 -p22,80,443,3389,8080 --rate=10000 -oJ scan_result.json

# 示例：Python 封装
python scripts/ip_scanner.py --target 192.168.0.0/16 --rate 50000
```

**关键参数**：
- `--rate`：发包速率，根据带宽调整
- `--excludefile`：排除IP列表（避免扫描敏感目标）
- `--banners`：是否抓取Banner

### 板块2：端口扫描（深度扫描）

**目标**：对存活IP进行精细端口扫描，识别开放服务

| 项目 | 说明 |
|:---|:---|
| 工具 | nmap（精确）/ rustscan（快速） |
| 输入 | 存活IP列表 |
| 输出 | IP:Port 映射 + 服务版本 |
| 端口范围 | 常用端口 / Top1000 / 全端口 |

```bash
# 常用端口扫描
nmap -sV -sC -p 22,80,443,3389,8080,8443 -iL alive_ips.txt -oX port_scan.xml

# 全端口快速扫描
rustscan -a 192.168.1.100 --ulimit 5000
```

**重点端口清单**：
| 端口 | 服务 | 用途 |
|:---|:---|:---|
| 22 | SSH | 远程登录部署 |
| 80/443 | HTTP/HTTPS | Web服务 |
| 3389 | RDP | Windows远程桌面 |
| 8080/8443 | HTTP-Alt | 应用服务 |
| 3306 | MySQL | 数据库 |
| 6379 | Redis | 缓存服务 |
| 27017 | MongoDB | 数据库 |

### 板块3：存活探测（验证扫描）

**目标**：验证IP是否真正可达、服务是否可用

| 项目 | 说明 |
|:---|:---|
| 工具 | ping / TCP Connect / HTTP Probe |
| 输入 | 扫描结果 |
| 输出 | 可达性状态 + 延迟 + 稳定性 |
| 频率 | 单次验证 / 周期性探测 |

```bash
# ICMP探测
fping -a -g 192.168.1.0/24 2>/dev/null > alive.txt

# HTTP存活探测
httpx -l urls.txt -status-code -title -tech-detect -o http_alive.json

# TCP连接验证
python scripts/alive_checker.py --hosts scan_result.json --timeout 3
```

**判断标准**：
- ICMP可达 → 基础存活
- TCP端口可连 → 服务存活
- HTTP响应200 → Web服务可用
- SSH握手成功 → 可部署节点

### 板块4：资产指纹识别（画像扫描）

**目标**：识别目标主机的操作系统、服务版本、框架指纹

| 项目 | 说明 |
|:---|:---|
| 工具 | nmap OS Detection / whatweb / wappalyzer |
| 输入 | 存活IP + 开放端口 |
| 输出 | OS类型、服务版本、Web框架、中间件 |
| 用途 | 匹配部署方案、评估算力能力 |

```bash
# 操作系统识别
nmap -O -sV 192.168.1.100

# Web指纹识别
whatweb http://192.168.1.100

# 批量指纹识别
python scripts/fingerprint.py --hosts alive.json --output assets.json
```

**输出示例**：
```json
{
  "ip": "192.168.1.100",
  "os": "Ubuntu 22.04",
  "arch": "x86_64",
  "cpu_cores": 4,
  "services": [
    {"port": 22, "service": "OpenSSH 8.9"},
    {"port": 80, "service": "nginx 1.24"}
  ],
  "deploy_compatible": true,
  "estimated_computing_power": "medium"
}
```

# 04_算力调度

> 核心任务：智能分配计算任务到各节点，最大化算力利用率

---

## 一、模块概述

算力调度模块是分布式算力矩阵的「大脑」，负责将计算任务合理分配到各个算力节点，实现负载均衡、故障转移和资源最大化利用。

## 二、调度架构

```
                     ┌────────────────────┐
                     │    任务提交入口     │
                     │  (API / CLI / Web)  │
                     └─────────┬──────────┘
                               │
                     ┌─────────▼──────────┐
                     │    调度中心         │
                     │  (Scheduler Core)   │
                     │                     │
                     │  ┌───────────────┐  │
                     │  │ 任务队列      │  │
                     │  │ (Task Queue)  │  │
                     │  └───────┬───────┘  │
                     │          │          │
                     │  ┌───────▼───────┐  │
                     │  │ 调度策略引擎  │  │
                     │  │ (Scheduler)   │  │
                     │  └───────┬───────┘  │
                     │          │          │
                     │  ┌───────▼───────┐  │
                     │  │ 节点池管理    │  │
                     │  │ (Node Pool)   │  │
                     │  └───────────────┘  │
                     └─────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │  Node A  │    │  Node B  │    │  Node C  │
        │  CPU:40% │    │  CPU:60% │    │  CPU:20% │
        │  RAM:2GB │    │  RAM:4GB │    │  RAM:8GB │
        └──────────┘    └──────────┘    └──────────┘
```

## 三、调度策略

### 3.1 负载均衡策略

| 策略 | 说明 | 适用场景 |
|:---|:---|:---|
| **轮询** (Round Robin) | 依次分配给每个节点 | 节点配置相近 |
| **加权轮询** (Weighted RR) | 按权重比例分配 | 节点配置差异大 |
| **最少连接** (Least Conn) | 分配给当前任务最少的节点 | 任务时长不均 |
| **资源感知** (Resource Aware) | 按实时CPU/内存余量分配 | 推荐默认策略 |
| **亲和性** (Affinity) | 相关任务分配到同一节点 | 有数据依赖 |

### 3.2 故障转移

```
节点心跳超时(30s) → 标记不健康 → 移出调度池 → 任务重新分配
                                                    │
节点恢复心跳 → 健康检查通过 → 重新加入调度池 ←──────┘
```

**转移策略**：
- 心跳超时 → 等待2次重试 → 确认失联 → 迁移任务
- 任务执行失败 → 记录错误 → 重试同节点1次 → 换节点重试
- 节点负载过高(>90%) → 暂停新任务分配 → 等待负载下降

### 3.3 优先级队列

| 优先级 | 说明 | 队列 |
|:---|:---|:---|
| P0 - 紧急 | 立即执行，可抢占 | high_priority |
| P1 - 高 | 优先执行 | normal |
| P2 - 普通 | 正常排队 | normal |
| P3 - 低 | 空闲时执行 | low_priority |

# 05_监控运维

> 核心任务：实时监控集群健康状态，及时告警，保障算力矩阵稳定运行

---

## 一、模块概述

监控运维模块是分布式算力矩阵的「眼睛和耳朵」，提供全方位的集群可观测性，确保每个节点的健康状态、资源使用、任务执行都在掌控之中。

## 二、监控架构

```
└──────────────────────────┬──────────────────────────────┘
                           │
```

## 三、监控维度

### 3.1 节点健康监控

| 指标 | 采集方式 | 阈值 | 告警级别 |
|:---|:---|:---|:---|
| 心跳状态 | Agent上报 | 超时90s | Critical |
| CPU使用率 | /proc/stat | >90% | Warning |
| 内存使用率 | /proc/meminfo | >85% | Warning |
| 磁盘使用率 | df | >90% | Critical |
| 网络连通性 | ping/tcp | 丢包>10% | Warning |
| 系统负载 | uptime | >CPU核数*2 | Warning |

### 3.2 任务执行监控

| 指标 | 说明 | 告警条件 |
|:---|:---|:---|
| 任务成功率 | 成功/总数 | <95% |
| 任务平均耗时 | 从开始到完成 | 超过预期2倍 |
| 队列堆积数 | 等待中的任务 | >1000 |
| 失败重试率 | 重试/总数 | >10% |
| 死信队列数 | 最终失败的任务 | >0 |

### 3.3 资源利用率监控

| 指标 | 说明 | 优化触发 |
|:---|:---|:---|
| 集群总算力 | 所有节点CPU总核数 | - |
| 算力利用率 | 已用/总量 | <30%（浪费） |
| 内存利用率 | 已用/总量 | 不均衡时 |
| 网络带宽 | 入/出流量 | 接近上限 |

## 四、告警系统

### 4.1 告警级别

| 级别 | 说明 | 通知方式 | 响应时间 |
|:---|:---|:---|:---|
| **Critical** | 严重故障，影响服务 | 电话+短信+微信 | 5分钟 |
| **Warning** | 潜在风险，需关注 | 微信+邮件 | 30分钟 |
| **Info** | 信息通知 | 邮件/日志 | 不限 |

### 4.2 告警规则

```yaml
alerts:
  - name: node_offline
    condition: "heartbeat_timeout > 90s"
    level: critical
    message: "节点 {{node_id}} ({{ip}}) 离线，最后心跳 {{last_heartbeat}}"
    
  - name: high_cpu
    condition: "cpu_usage > 90% for 5min"
    level: warning
    message: "节点 {{node_id}} CPU持续高负载 {{cpu_usage}}%"
    
  - name: disk_full
    condition: "disk_usage > 90%"
    level: critical
    message: "节点 {{node_id}} 磁盘即将满 {{disk_usage}}%"
    
  - name: task_failure_spike
    condition: "task_failure_rate > 10% in 10min"
    level: warning
    message: "任务失败率异常升高 {{failure_rate}}%"
    
  - name: queue_backlog
    condition: "queue_size > 1000"
    level: warning
    message: "任务队列堆积 {{queue_size}} 个任务"
```

### 4.3 通知渠道

| 渠道 | 配置 | 适用级别 |
|:---|:---|:---|
| 微信/企业微信 | Webhook URL | Warning + Critical |
| 邮件 | SMTP配置 | 所有级别 |
| 短信 | API接口 | Critical |
| Telegram | Bot Token | Warning + Critical |
| 飞书 | Webhook | 所有级别 |

## 五、日志系统

### 5.1 日志分类

| 日志类型 | 来源 | 存储 | 保留 |
|:---|:---|:---|:---|
| 节点日志 | Agent运行日志 | 本地+中心 | 30天 |
| 任务日志 | 任务执行输出 | 中心 | 60天 |
| 调度日志 | 调度决策记录 | 中心 | 90天 |
| 审计日志 | 操作记录 | 中心 | 365天 |
| 系统日志 | 平台运行日志 | 中心 | 30天 |

### 5.2 日志查询

```bash
# 查看某节点最近日志
python scripts/monitor.py logs --node node-001 --tail 100

# 搜索错误日志
python scripts/monitor.py logs --level error --since "1h ago"

# 查看某任务的执行日志
python scripts/monitor.py logs --task task-uuid-001

# 导出日志
python scripts/monitor.py logs --since "2026-02-01" --export logs.csv
```

## 六、运维操作

### 6.1 集群状态总览

```bash
# 查看集群状态
python scripts/monitor.py status

║ CPU总核数：200   已用：120   利用率：60%          ║

║ 今日失败：5      成功率：99.96%                   ║

```

### 6.2 节点运维

```bash
# 节点排空（停止分配新任务，等待现有任务完成）
python scripts/monitor.py drain --node node-001

# 节点重启Agent
python scripts/monitor.py restart --node node-001

# 节点移除
python scripts/monitor.py remove --node node-004

# 批量健康检查
python scripts/monitor.py health-check --all
```

### 6.3 定期巡检

```bash
# 每日巡检报告
python scripts/daily_report.py

# 输出：
# - 节点在线率
# - 任务完成率
# - 资源利用率趋势
# - 告警统计
# - 异常节点列表
```

## 九、配置文件

```yaml
  enabled: true
  check_interval: 30
  silence_duration: 3600     # 相同告警静默时间
  channels:
    wechat:
      enabled: true
      webhook: "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx"

      enabled: true
      smtp_host: "smtp.example.com"
      smtp_port: 465
      sender: "monitor@example.com"
      receivers: ["admin@example.com"]
    telegram:
      enabled: false
      bot_token: ""
      chat_id: ""

logs:
  storage: file              # file / elasticsearch
  path: ./logs/
  retention_days: 30
  max_size_mb: 500
  
dashboard:
  enabled: true
  bind: "0.0.0.0:3000"
  auth: true
  username: admin
  password_env: DASHBOARD_PASSWORD

reports:
  daily:
    enabled: true
    time: "09:00"
    send_to: wechat
  weekly:
    enabled: true
    day: monday
    time: "09:00"
```
