  contents: # GitHub（fnvtk）同步到 NAS Git 方案与每日 1 点计划

> 以 GitHub 为主，NAS 做备份/恢复/本地调取；每天凌晨 1 点自动从 GitHub 拉取最新到 NAS。

---

## 1. 目标与原则

| 项目 | 说明 |
|:---|:---|
| **主源** | GitHub（fnvtk 账号下所有仓库） |
| **备份/本地** | NAS 上的 Git 镜像（`/volume1/git/github/fnvtk/`） |
| **同步方向** | 仅 GitHub → NAS（单向拉取，不向 GitHub 回写） |
| **自动策略** | 每天凌晨 1:00 执行一次全量同步（新仓 clone --mirror，已有仓 fetch --all --prune） |

---

## 2. 使用的脚本（安全版）

- **脚本名**：`sync_github_simple_fixed.sh`（卡若AI 仓库内）  
- **NAS 上的部署路径**：`/volume1/scripts/git_sync/sync_github_simple.sh`（建议只保留此唯一入口，避免误跑旧版）

脚本已具备：

- 单实例锁（防止多实例叠加导致 NAS 负载暴增）
- 最大 API 页数限制（MAX_PAGE=200）
- API 报错时立即退出
- Token 从 `.secrets.env` 读取，不写死在脚本里
- 同步范围：当前 Token 对应用户（fnvtk）的**所有**仓库

---

## 3. NAS 上部署步骤

### 3.1 准备目录与密钥

```bash
# SSH 登录 NAS（示例）
ssh fnvtk@192.168.1.201

# 创建目录
sudo mkdir -p /volume1/scripts/git_sync
sudo mkdir -p /volume1/logs/git_sync
sudo chown -R fnvtk:users /volume1/scripts/git_sync /volume1/logs/git_sync

# 创建密钥文件（不要提交到 Git）
nano /volume1/scripts/git_sync/.secrets.env
```

在 `.secrets.env` 中写入（替换为你的 GitHub Personal Access Token）：

```bash
export GITHUB_TOKEN="ghp_xxxxxxxxxxxxxxxxxxxx"
export GITHUB_USER="fnvtk"
```

保存后：

```bash
chmod 600 /volume1/scripts/git_sync/.secrets.env
```

### 3.2 部署脚本

- 将本仓库中的 `scripts/sync_github_simple_fixed.sh` 拷贝到 NAS：
  - 目标路径：`/volume1/scripts/git_sync/sync_github_simple.sh`
- 若 NAS 上仍有旧版 `sync_github_simple.sh`，建议先改名为 `sync_github_simple_OLD.sh`，再放新脚本，避免 cron 误调旧版。
- 赋予执行权限：

```bash
chmod +x /volume1/scripts/git_sync/sync_github_simple.sh
```

### 3.3 设置每天凌晨 1 点自动同步

用 crontab 配置（以 fnvtk 用户执行）：

```bash
crontab -e
```

增加一行：

```cron
0 1 * * * /volume1/scripts/git_sync/sync_github_simple.sh
```

表示每天 01:00 执行一次。保存退出。

检查是否生效：

```bash
crontab -l
```

## 4. 镜像目录与 NAS Git Server 的关系

- **镜像根目录**：`/volume1/git/github`  
  - fnvtk 的仓库会落在：`/volume1/git/github/fnvtk/<repo>.git`（裸仓）。
- **Container Manager 中的「Git Server」**（如 open.quwanzhi.com:5200 里看到的 admin / fnvtk / guest）：
  - 若 Git Server 套件使用的仓库根目录就是 `/volume1/git` 或已配置指向 `/volume1/git/github`，则这些裸仓会被识别为「fnvtk」下的仓库，无需额外操作。
  - 若 Git Server 使用独立存储路径，则需要在 Git Server 中把「fnvtk」对应的仓库目录指向 `MIRROR_ROOT/fnvtk`，或另行用脚本把 mirror 推送到 Git Server 的 URL（需 Git Server 支持 SSH/HTTP 推送）。

日常使用建议：**以 GitHub 为主开发与推送，NAS 仅作定时拉取的备份与本地克隆源**。

## 6. 注意事项（避免再次负载暴增）

- **不要**在 NAS 上同时运行多个同步脚本实例；当前脚本已带单实例锁，重复执行会自动跳过。
- **不要**恢复使用未限制 API 页数、无错误退出的旧版脚本。
- 若需调整同步账号或 Token，只改 `/volume1/scripts/git_sync/.secrets.env`，不要改脚本内的默认值（脚本会 source 该文件）。

> NAS 上的 Git 是 GitHub 的镜像备份，以 **GitHub 为主**，NAS 每日自动同步。本机日常在 GitHub 开发，需要时从 NAS 克隆或恢复。

## 1. 原则

| 角色 | 说明 |
|:---|:---|
| **GitHub** | 主源，日常 push/pull 在 GitHub |
| **NAS** | 备份 + 本地克隆源，每天 01:00 自动从 GitHub 拉取 |
| **本机 Mac** | 从 GitHub 或 NAS 都可以 clone；推荐平时用 GitHub，断网/恢复时用 NAS |

## 3. 本机日常怎么用

- **正常开发**：在本机 clone 的是 **GitHub** 的地址（如 `git clone https://github.com/fnvtk/my.git`），日常 push/pull 对 GitHub 即可；NAS 会自动同步，无需本机操作。
- **只用 NAS 备份**：不打算从 NAS clone 的话，不用做任何事，NAS 每天会自动拉取 GitHub 最新。
- **从 NAS 恢复/离线用**：在本机执行上面第 2 节的 `git clone fnvtk@192.168.1.201:/volume1/git/github/fnvtk/xxx.git`，得到的就是 NAS 上的镜像副本。

# 将 fnvtk 的 GitHub 全部仓库同步到 CKB NAS 上的 Gitea（open.quwanzhi.com:3000）

while [ $# -gt 0 ]; do

if [ -z "$GITEA_TOKEN" ]; then

get_repos() {
  if [ -n "$SINGLE_REPO" ]; then
    echo "$SINGLE_REPO"
    return
  fi
  if [ -z "$GITHUB_TOKEN" ]; then
    echo "错误: 全量同步需要 GITHUB_TOKEN（GitHub Settings → Developer settings → Personal access tokens）"
    exit 1
  fi
  curl -s -H "Authorization: token $GITHUB_TOKEN" \

# 在 Gitea 创建仓库（若不存在）
gitea_create_repo() {
  local repo_name="$1"
  local exists
  exists=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Authorization: token ${GITEA_TOKEN}" \
    "${GITEA_BASE}/api/v1/repos/${GITEA_USER}/${repo_name}" 2>/dev/null) || true
  if [ "$exists" = "200" ]; then
    return 0
  fi
  echo "  在 Gitea 创建仓库: ${repo_name}"
  curl -s -X POST \
    -H "Authorization: token ${GITEA_TOKEN}" \
    -H "Content-Type: application/json" \
    -d "{\"name\":\"${repo_name}\",\"private\":false}" \
    "${GITEA_BASE}/api/v1/user/repos" >/dev/null || true
}

# 单个仓库：从 GitHub mirror 拉取，推送到 Gitea
sync_one() {
  local repo_name="$1"
  local gh_url="https://github.com/${GITHUB_USER}/${repo_name}.git"
  if [ -n "$GITHUB_TOKEN" ]; then
    gh_url="https://${GITHUB_TOKEN}@github.com/${GITHUB_USER}/${repo_name}.git"
  fi
  local gitea_url="${GITEA_BASE/https:\/\//https:\/\/${GITEA_TOKEN}@}"
  gitea_url="${gitea_url/http:\/\//http:\/\/${GITEA_TOKEN}@}"
  gitea_url="${gitea_url%/}/${GITEA_USER}/${repo_name}.git"

  echo "同步: ${repo_name}"
  gitea_create_repo "$repo_name"

  mkdir -p "$WORK_DIR"
  cd "$WORK_DIR"
  if [ -d "${repo_name}.git" ]; then
    cd "${repo_name}.git"
    git fetch --all --prune
    git fetch --tags --prune
  else
    git clone --mirror "$gh_url" "${repo_name}.git"
    cd "${repo_name}.git"
  fi
  git push --mirror "$gitea_url" || { echo "  警告: ${repo_name} 推送失败"; return 1; }
  echo "  完成: ${repo_name}"
}

mkdir -p "$WORK_DIR"

done < <(get_repos)

echo "全部同步完成。Gitea: ${GITEA_BASE}/${GITEA_USER}"

## 一、架构与流向

| 方向 | 说明 | 实现方式 |
|:-----|:-----|:---------|
| **GitHub → Gitea** | 主向同步，保持 Gitea 与 GitHub 一致 | 脚本 `sync_github_to_gitea.sh` + 定时 / GitHub Actions |
| **Gitea → GitHub** | 可选，仅在「以 Gitea 为主」的少数仓库使用 | Gitea Webhook 或本地 push 后手动/脚本推送到 GitHub |

**建议**：默认以 **GitHub 为源真相**，Gitea 为镜像；双向仅对个别在 NAS 上也会改动的仓库配置。

## 二、前置条件

- **CKB NAS 上 Gitea 已部署**：见 `references/CKB_NAS_Gitea_类GitHub访问.md`，地址 `http://open.quwanzhi.com:3000`（或内网 `http://192.168.1.201:3000`）。
- **Gitea 令牌**：Gitea 登录 → 设置 → 应用 → 生成新令牌，权限勾选 `repo`（或按需），保存为 **GITEA_TOKEN**。
- **GitHub 令牌（全量同步必填）**：GitHub → Settings → Developer settings → Personal access tokens，生成带 `repo` 的令牌，用于拉取/列出 fnvtk 下所有仓库，保存为 **GITHUB_TOKEN**。

## 四、自动触发：GitHub 有更新就推到 Gitea

### 4.1 使用 GitHub Actions（推荐）

仓库根目录已提供 workflow：`.github/workflows/sync_github_to_gitea.yml`。

- **触发方式**：  
  - 定时：每 6 小时一次；  
  - 手动：Actions 页选择 “Sync GitHub to CKB NAS Gitea” → Run workflow；  
  - 可选：对 `main`/`master` 的 push 触发（已排除仅改 .md 和本 workflow 的提交）。
- **仓库 Secrets 配置**：  
  - **GITEA_TOKEN**（必填）：Gitea 令牌。  
  - **GITHUB_TOKEN**（全量同步必填）：若用默认 `GITHUB_TOKEN` 只能访问当前仓库；要同步 fnvtk 下**全部**仓库，请在仓库 Settings → Secrets 中新增 **GITHUB_TOKEN**（或 **GH_PAT**），值为 GitHub PAT，并在脚本/workflow 中引用该 secret。

若 workflow 中改为使用 `GH_PAT` 列仓库，则在 “Sync GitHub → Gitea” 步骤里设置：

```yaml
env:
  GITHUB_TOKEN: ${{ secrets.GH_PAT }}
```

并确保该仓库的 Secrets 中有 **GH_PAT**。

### 4.2 本机或 NAS 上定时（cron）

在能同时访问 GitHub 与 Gitea 的机器上：

```bash
# 每 6 小时执行一次全量同步
0 */6 * * * GITHUB_TOKEN=xxx GITEA_TOKEN=xxx bash /path/to/sync_github_to_gitea.sh >> /var/log/github_gitea_sync.log 2>&1
```

将 `GITHUB_TOKEN`、`GITEA_TOKEN` 和脚本路径替换为实际值；敏感信息建议放环境文件或密钥管理，勿写进 crontab 明文。

## 五、双向同步说明

- **主向**：**GitHub → Gitea** 已由上述脚本 + 定时/Actions 覆盖，保证 GitHub 一有更新就推到 Gitea。
- **反向**：**Gitea → GitHub** 仅在部分仓库需要时再开：
  - **方式一**：在 Gitea 仓库设置 → Web 钩子 → 添加 “Gitea” 或 “HTTP” 钩子，指向自建服务：该服务在收到 push 后执行 `git push` 到对应 GitHub 仓库。需自行处理冲突与权限。
  - **方式二**：在 NAS 或本机对「在 Gitea 上也会改」的仓库，配置双 remote（origin=Gitea、github=GitHub），提交后先 push Gitea 再 `git push github main`，或写小脚本在 push Gitea 后自动 push GitHub。

建议：大部分仓库仅做 **GitHub → Gitea** 单向镜像；需要双向的仓库单独配置并约定「以谁为主、冲突如何解决」。

> **维护**：金仓

---

## 一、脚本 vs 钩子

| 方式 | 触发 | 方向 | 说明 |
|:-----|:-----|:-----|:-----|
| **定时脚本**（cron） | 每 30 分钟 | GitHub → Gitea | NAS 上执行 `sync_github_to_gitea.sh`，拉取 GitHub 全仓/指定仓并推送到存客宝 Gitea。 |
| **GitHub Web 钩子** | GitHub 有 push 时 | GitHub → Gitea | GitHub 向 NAS 上的接收端发 POST，接收端执行同步脚本（实时）。 |
| **Gitea Web 钩子** | Gitea 有 push 时 | Gitea → 外部 | Gitea 向指定 URL 发 POST（通知用）；若要做 Gitea→GitHub 反向同步，需接收端再推送到 GitHub。 |

---

## 二、是否冲突？结论：不冲突，用锁兜底

- **脚本与 GitHub 钩子**：做的是同一件事（GitHub → Gitea），只是触发不同（定时 vs 事件）。可能同时发生（例如刚 push 完又赶上 cron），**不会逻辑冲突**，但并发跑同一仓库可能造成重复拉取/推送。
- **处理方式**：同步脚本已加**全局锁**（同一时间只允许一个同步进程）。钩子或 cron 触发时，若脚本已在跑，后到的会等待锁或超时跳过，**不会两个进程同时写同一仓库**。
- **Gitea 钩子**：若只做「GitHub → Gitea」单向，**不需要在 Gitea 配置钩子**。若要做「Gitea → GitHub」反向，Gitea 钩子只负责「通知接收端」，由接收端去 push 到 GitHub，与脚本（GitHub→Gitea）方向相反，**不冲突**。

---

## 三、最佳方案（推荐）

1. **主向：GitHub → Gitea（实时 + 兜底）**
   - **实时**：在 **GitHub** 仓库或组织里配置 Web 钩子，Payload URL 指向 NAS 上的 webhook 接收端（需 NAS 有可从外网访问的地址，如 frp）；触发时只同步该仓库（或按约定同步全仓）。
   - **兜底**：保留 **cron 每 30 分钟** 全量同步，防止钩子漏触发或网络问题。
   - **规则**：脚本带锁，钩子与 cron 都只调同一脚本，不会并发冲突。

2. **Gitea 钩子（按需）**
   - 若只需「GitHub 为源、Gitea 为镜像」，**不必在 Gitea 配置钩子**。
   - 若需「在 Gitea 的修改也推到 GitHub」：
     - 在 Gitea **用户设置 → Web 钩子** 添加「添加 Web 钩子」；
     - URL 填接收端地址，触发事件选「推送」等；
     - 接收端收到后，根据 payload 找到对应仓库并执行 `git push` 到 GitHub（需在接收端配置 GitHub 权限）。

3. **约定规则（固定）**
   - **源真相**：默认以 **GitHub 为准**，Gitea 为镜像；反向（Gitea→GitHub）仅对明确需要的仓库开启。
   - **只用一个同步脚本**：所有同步（cron 或钩子触发）都调用 `sync_github_to_gitea.sh`（可带 `--repo xxx`），脚本内用同一把锁。
   - **不在 Gitea 侧做「镜像仓库」与脚本重复**：若用 Gitea 的「从 URL 迁移/镜像」功能，就不要再对同一仓库用本脚本，二选一，避免双写。

---

## 五、脚本锁规则（实现层面）

- 同步脚本使用**全局锁目录** `$WORK_DIR.lock`（如 `/tmp/github_gitea_sync.lock`），通过 `mkdir` 原子性获取锁。
- 获取不到锁时，每 5 秒重试，超过 `SYNC_LOCK_WAIT` 秒（默认 300）则退出，不执行本次同步。
- 脚本正常或异常退出时用 `trap EXIT` 释放锁。
- **结论**：cron 与 webhook 可同时配置，两者都触发同一脚本，由锁保证同一时刻只跑一个同步，无冲突。

class SyncHandler(BaseHTTPRequestHandler):
    def do_POST(self):
        repo = None
        try:
            length = int(self.headers.get("Content-Length", 0))
            if length:
                body = self.rfile.read(length).decode("utf-8", errors="ignore")

                    data = json.loads(body)

                    pass

            pass

        if repo:

        env["SYNC_WORK_DIR"] = os.environ.get("SYNC_WORK_DIR", "/tmp/github_gitea_sync")

            ["/bin/bash"] + cmd,
            cwd=SCRIPT_DIR,
            env=env,
            stdout=subprocess.DEVNULL,
            stderr=subprocess.DEVNULL,
        )

        self.send_header("Content-Type", "text/plain; charset=utf-8")
        self.end_headers()
        self.wfile.write(b"ok")

        if self.path in ("/", "/sync", "/health"):
            self.send_response(200)
            self.send_header("Content-Type", "text/plain; charset=utf-8")
            self.end_headers()

        else:
            self.send_response(404)
            self.end_headers()

        sys.stderr.write("[webhook] %s\n" % (fmt % args))
