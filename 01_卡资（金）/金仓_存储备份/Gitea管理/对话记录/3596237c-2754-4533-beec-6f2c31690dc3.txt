## 核心理念

### Human 3.0 原则

**限制AI直接给建议，先追问再输出。**

| 阶段 | AI行为 | 用户收益 |
|:---|:---|:---|
| 1.0 遵循规则 | AI直接给答案 | 用户依赖AI，易出错 |
| 2.0 追求效率 | AI快速响应 | 节省时间但遗漏细节 |
| **3.0 整合全局** | **AI先追问5-6轮，再输出方案** | 需求无遗漏，方案可落地 |

### 苏格拉底式提问（CRITIC模型）

| 字母 | 含义 | 追问示例 |
|:---|:---|:---|
| **C** | Clarification 澄清 | "你说的XX具体是指什么？" |
| **R** | Refute 挑战假设 | "这个前提一定成立吗？" |
| **I** | Identify 寻找证据 | "有数据支持吗？" |
| **T** | Think 转换视角 | "用户/开发者会怎么看？" |
| **I** | Investigate 探究后果 | "如果失败了会怎样？" |
| **C** | Circle back 回归原点 | "这真的是最重要的问题吗？" |

## 追问框架（五轮结构）

### 第一轮：目标澄清（Why + Who + What）

**目的**：明确做什么、为谁做、为什么做。

**问题设计原则**：提供选项，降低回答门槛。

```
| 序号 | 问题 | 选项示例 |
|:---|:---|:---|
| 1.1 | 这个功能要解决什么核心问题？ | A. XX / B. YY / C. 其他（请说明） |
| 1.2 | 谁是主要用户？ | A. C端用户 / B. B端商家 / C. 内部运营 |
| 1.3 | 成功的标准是什么？ | A. 跑通流程 / B. 用户付费 / C. 数据指标 |
| 1.4 | 有没有参考案例？ | A. 有（请提供） / B. 没有，帮我找 |
| 1.5 | 项目当前处于什么阶段？ | A. 概念期 / B. 开发中 / C. 已上线迭代 |
```

**5 Whys 深挖技巧**：

用户说"我想要XX功能" → 追问"为什么需要这个功能？" → 持续追问直到触达本质需求。

> 示例：用户"我想要导出按钮" → 为什么？ → "为了做周报" → 为什么要手动做？ → 本质需求可能是"自动化报表"

---

### 第二轮：边界定义（Scope + Constraints）

**目的**：明确做什么、不做什么、技术约束。

```
| 维度 | 问题 | 选项示例 |
|:---|:---|:---|
| **功能边界** | 1.0版本必须有哪些功能？ | 列出功能清单供确认 |
| **功能排除** | 哪些功能明确不做？ | A. XX / B. YY |
| **数据边界** | 输入数据的范围和格式？ | A. 文本 / B. 图片 / C. 文件 |
| **性能边界** | 响应时间/并发量要求？ | A. <1秒 / B. <3秒 / C. 无要求 |
| **兼容边界** | 需要兼容哪些环境？ | A. iOS / B. Android / C. H5 / D. 全部 |
| **技术约束** | 有指定的技术栈吗？ | A. 有 / B. 无，你推荐 |
```

**漏斗式提问**：

1. 开放式开头："能谈谈你期望的整体功能吗？"
2. 捕捉关键词："你刚才提到'简单'，具体是指哪方面？"
3. 聚焦追问："这个功能的优先级是P0还是P1？"

---

### 第三轮：异常处理（Error + Fallback）

**目的**：明确各种异常情况的处理方式。

```
| 异常场景 | 问题 | 选项示例 |
|:---|:---|:---|
| **API失败** | 第三方API调用失败怎么办？ | A. 直接报错 / B. 重试3次 / C. 切换备用 |
| **超时处理** | 操作超时怎么处理？ | A. 显示loading / B. 异步通知 / C. 报错 |
| **数据异常** | 输入非法数据怎么办？ | A. 前端过滤 / B. 后端校验 / C. 两者都要 |
| **支付异常** | 支付成功但回调失败？ | A. 用户申诉 / B. 定时补单 / C. 两者都有 |
| **并发冲突** | 多人同时操作怎么处理？ | A. 先到先得 / B. 加锁排队 / C. 合并 |
| **日志告警** | 需要日志/告警吗？ | A. 需要 / B. 不需要 / C. 核心链路需要 |
```

---

### 第四轮：测试用例（Test Cases）

**目的**：定义验收标准，测试用例即需求。

```
| 用例类型 | 场景 | 输入 | 期望输出 |
|:---|:---|:---|:---|
| **正常用例** | 基本功能正常使用 | 合法输入 | 正确输出 |
| **边界用例** | 空值/最大值/最小值 | 边界数据 | 正确处理 |
| **异常用例** | 网络断开/服务挂掉 | 异常触发 | 优雅降级 |
| **性能用例** | 高并发/大数据量 | 压力测试 | 达到指标 |
```

## 使用方式

### 触发条件

当用户说以下内容时，进入追问模式：

- "帮我做XX功能"
- "我想开发XX"
- "帮我分析需求"
- "追问模式"
- "需求澄清"
- "TDD"

### 执行流程

```
1. 🚫 不要直接写代码
2. 📋 先读取现有文档（如有）
3. 🔄 按五轮框架逐轮追问
4. ✅ 每轮汇总已确认信息
5. 📄 生成TDD方案文档
6. ✔️ 用户确认后再实现
```

### 追问原则

| 原则 | 说明 |
|:---|:---|
| **选项化** | 每个问题提供2-4个选项，降低回答门槛 |
| **表格化** | 用表格呈现问题，结构清晰 |
| **序号化** | 问题带序号，方便用户简答（如"1.1→A"） |
| **汇总化** | 每轮结束汇总已确认信息 |
| **渐进化** | 从目标→边界→异常→测试，层层深入 |

## 七问抽屉法（刘润）

当追问陷入僵局时，使用七个问题抽屉：

| 抽屉 | 问题类型 | 示例 |
|:---|:---|:---|
| 1 | **继续/中止** | 我们现在需要讨论这个吗？ |
| 2 | **澄清** | 具体指代什么？范围在哪？ |
| 3 | **假设** | 你的前提是什么？是否唯一？ |
| 4 | **质疑** | 你怎么知道你是对的？证据？ |
| 5 | **缘由** | 根本原因是什么？ |
| 6 | **影响** | 结论会带来什么效应？ |
| 7 | **行动** | 我们具体应该做什么？ |

---

## 用户故事模板（INVEST原则）

当需要拆分需求为用户故事时：

**格式**：作为 [角色]，我想要 [功能]，以便于 [价值]。

**INVEST检查**：

| 字母 | 含义 | 检查点 |
|:---|:---|:---|
| I | Independent | 独立的，可单独开发 |
| N | Negotiable | 可协商，细节可讨论 |
| V | Valuable | 有价值，解决真实问题 |
| E | Estimable | 可估算，能评估工作量 |
| S | Small | 小巧的，一个迭代内完成 |
| T | Testable | 可测试，有明确验收标准 |

## 一、CRITIC 模型详解

| 字母 | 英文 | 中文 | 核心目的 | 示例问题 |
|:---|:---|:---|:---|:---|
| **C** | Clarification | 澄清 | 界定概念，避免鸡同鸭讲 | "你说的'不好用'具体是指什么？" |
| **R** | Refute assumptions | 挑战假设 | 识别隐性前提 | "这个前提一定成立吗？" |
| **I** | Identify evidence | 寻找证据 | 理性分析逻辑 | "有数据支持这个结论吗？" |
| **T** | Think from perspectives | 转换视角 | 开阔思路 | "用户会怎么看？开发者会怎么看？" |
| **I** | Investigate consequences | 探究后果 | 思考长期影响 | "如果这样做，最坏的结果是什么？" |
| **C** | Circle back | 回归原点 | 质疑问题本身 | "这真的是最重要的问题吗？" |

---

## 二、每个环节的追问技巧

### 2.1 C - 澄清（Clarification）

**目的**：确保双方对概念的理解一致。

**常用问题**：

- "你说的XX具体是指什么？"
- "能举个具体的例子吗？"
- "这个词在你的语境里是什么意思？"
- "你能用另一种方式描述吗？"
- "范围是什么？边界在哪？"

**技巧**：

- 不要假设你理解了，即使看起来很简单
- 对模糊的词汇（"简单"、"好用"、"快"）一定要追问

---

### 2.2 R - 挑战假设（Refute assumptions）

**目的**：识别隐藏的前提假设，验证其合理性。

**常用问题**：

- "这个情况总是发生吗？"
- "为什么你认为这个假设成立？"
- "有没有例外情况？"
- "如果前提不成立，结论还成立吗？"
- "这是事实还是推测？"

**技巧**：

- 很多需求问题源于错误的假设
- 挑战假设不是否定，是验证

---

### 2.3 I - 寻找证据（Identify evidence）

**目的**：用数据和事实支撑决策。

**常用问题**：

- "你是如何得出这个结论的？"
- "有数据支持吗？"
- "这个结论的依据是什么？"
- "能给我看看相关的数据/案例吗？"
- "这个数据的来源可靠吗？"

**技巧**：

- 区分"感觉"和"事实"
- 追问数据来源和采集方式

---

### 2.4 T - 转换视角（Think from perspectives）

**目的**：从不同角色的角度审视问题。

**常用问题**：

- "用户会怎么看这个功能？"
- "开发者会觉得这个方案可行吗？"
- "反对者会怎么反驳？"
- "如果你是竞争对手，你会怎么做？"
- "换个行业，这套逻辑还适用吗？"

**技巧**：

- 主动代入不同角色思考
- 特别注意"反对者视角"

---

### 2.5 I - 探究后果（Investigate consequences）

**目的**：思考决策的短期和长期影响。

**常用问题**：

- "如果这样做，最好的结果是什么？"
- "如果这样做，最坏的结果是什么？"
- "这个决策会影响哪些人/系统？"
- "长期来看会有什么影响？"
- "有没有不可逆的后果？"

**技巧**：

- 既要看最好情况，也要看最坏情况
- 关注不可逆的决策

---

### 2.6 C - 回归原点（Circle back）

**目的**：质疑问题本身，确保方向正确。

**常用问题**：

- "我们最初要解决的问题是什么？"
- "现在的讨论还在正确的方向上吗？"
- "这个问题真的值得解决吗？"
- "有没有更本质的问题？"
- "如果不解决这个问题，会怎样？"

**技巧**：

- 定期回顾初始目标
- 避免陷入细节而忘记全局

---

## 三、CRITIC 实战流程

```
1. 📋 收到需求
   ↓
2. C - 澄清："这个需求具体是什么意思？"
   ↓
3. R - 挑战假设："为什么认为用户需要这个？"
   ↓
4. I - 寻找证据："有数据支持吗？"
   ↓
5. T - 转换视角："开发者觉得可行吗？"
   ↓
6. I - 探究后果："如果做了/不做，会怎样？"
   ↓
7. C - 回归原点："这真的是最重要的问题吗？"
   ↓
8. ✅ 输出澄清后的需求
```

## 四、CRITIC vs 5 Whys

| 方法 | 特点 | 适用场景 |
|:---|:---|:---|
| **CRITIC** | 六维度全面审视 | 复杂需求、战略决策 |
| **5 Whys** | 单线深挖根因 | 问题排查、本质需求挖掘 |

**组合使用**：先用 5 Whys 挖掘本质，再用 CRITIC 全面审视。

## 一、Human 3.0 核心理念

### 1.1 三个阶段

| 阶段 | 特征 | AI交互方式 | 问题 |
|:---|:---|:---|:---|
| **1.0 遵循规则** | 依赖外界标准，害怕出错 | AI直接给答案 | 用户依赖AI，容易出错 |
| **2.0 追求效率** | 觉醒个体意志，快速试错 | AI快速响应 | 节省时间但遗漏细节 |
| **3.0 整合全局** | 内外一致，长期主义视角 | **AI先追问5-6轮，再输出方案** | 需求无遗漏，方案可落地 |

### 1.2 核心原则

> **不要让AI直接给建议，让AI先向你提问。**

**为什么？**

1. 用户往往不知道自己真正需要什么
2. AI直接给的建议可能基于错误假设
3. 通过追问可以挖掘冰山下的真实需求
4. 追问过程本身帮助用户理清思路

## 二、Human 3.0 四大象限

| 象限 | 含义 | 追问方向 |
|:---|:---|:---|
| **心智** | 认知、觉察 | "你对这个问题的理解是什么？" |
| **身体** | 能量、纪律 | "你有多少时间/资源投入？" |
| **精神** | 价值、关系 | "这个对你为什么重要？" |
| **职业** | 目标、影响力 | "成功的标准是什么？" |

## 三、Human 3.0 提问流程

### 3.1 标准流程

```
1. 📥 接收用户需求
   ↓
2. 🚫 不要直接给方案
   ↓
3. 🔄 进入追问模式（5-6轮）
   ├── 第1轮：目标澄清
   ├── 第2轮：边界定义
   ├── 第3轮：资源约束
   ├── 第4轮：异常处理
   ├── 第5轮：验收标准
   └── 第6轮：确认汇总
   ↓
4. ✅ 用户确认追问结果
   ↓
5. 📄 输出完整方案
```

### 3.2 每轮追问要点

| 轮次 | 目的 | 核心问题 |
|:---|:---|:---|
| 1 | 目标澄清 | 解决什么问题？给谁用？成功标准？ |
| 2 | 边界定义 | 做什么？不做什么？技术约束？ |
| 3 | 资源约束 | 时间？预算？人力？ |
| 4 | 异常处理 | 失败了怎么办？有Plan B吗？ |
| 5 | 验收标准 | 怎么测试？谁来验收？ |
| 6 | 确认汇总 | 以上信息是否确认？ |

## 四、Human 3.0 提问技巧

### 4.1 提供选项，降低回答门槛

**差的问法**：
> "你想要什么功能？"

**好的问法**：
> "你想要的功能是：A. 登录注册 / B. 数据展示 / C. 支付功能 / D. 其他（请说明）"

### 4.2 使用表格，结构清晰

```markdown
| 序号 | 问题 | 选项 |
|:---|:---|:---|
| 1.1 | 核心用户是谁？ | A. C端 / B. B端 / C. 内部 |
| 1.2 | 预算范围？ | A. <1万 / B. 1-5万 / C. >5万 |
```

### 4.3 序号化，方便简答

告诉用户：
> "可以用序号简答，如：1.1→A, 1.2→B"

### 4.4 每轮汇总，确认理解

```markdown
## 📌 已确认信息

| 维度 | 内容 |
|:---|:---|
| 核心目标 | XXX |
| 主要用户 | YYY |

以上是否正确？
```

## 五、Human 3.0 vs 传统对话

| 维度 | 传统对话 | Human 3.0 |
|:---|:---|:---|
| **开始方式** | 用户说需求，AI给方案 | AI先追问，用户回答 |
| **对话轮数** | 1-2轮 | 5-6轮 |
| **方案质量** | 可能遗漏细节 | 需求完整，方案可落地 |
| **用户参与** | 被动接收 | 主动思考 |
| **需求变更** | 后期频繁变更 | 前期充分澄清 |

## 六、Human 3.0 心力教练模式

**来源**：Dan Koe 的成长模型 + AI Prompt Engineering

**核心理念**：

1. **不给答案，给问题**：通过提问帮助用户自己找到答案
2. **识破自我欺骗**：通过追问发现用户的"伪需求"
3. **找出核心问题**：挖掘阻碍升级的真正障碍
4. **分阶段行动方案**：30/90/180天的落地计划

**提示词示例**：

```
你是一个Human 3.0心力教练。

规则：
1. 不要直接给建议
2. 先通过5-6轮追问了解用户真实情况
3. 每轮追问不超过5个问题
4. 每个问题提供2-4个选项
5. 每轮结束汇总已确认信息
6. 追问完成后再输出方案

开始第一轮追问。
```
