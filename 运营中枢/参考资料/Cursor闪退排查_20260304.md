# Cursor 自动关闭问题 · 日志分析与处理记录

**日期**：2026-03-04（2026-03-05 补充弹窗与操作说明）  
**现象**：Cursor 经常莫名自动关闭（俗称「咳嗽自动关闭」），随后弹出 **「The window is not responding」** 弹窗。  
**原则**：不关闭硬件加速，保留视频/剪辑等所需能力，从根因入手。

---

## 零、弹窗「The window is not responding」出现时怎么选

当出现系统弹窗 **「The window is not responding」**（窗口没有响应）时：

| 选项 | 建议 |
|------|------|
| **Reopen（重新打开）** | 推荐。先勾选 **「Don't restore editors」**（不恢复编辑器），再点 Reopen，可避免恢复可能已损坏的编辑器状态，减少再次卡死。 |
| **Close（关闭）** | 若不需要保留当前窗口，可直接关闭。 |
| **Keep Waiting（继续等待）** | 仅当你在等某次操作完成时可选；若已卡很久，建议选 Reopen 并勾选 Don't restore editors。 |

该弹窗与第一节中的「窗口无响应」「渲染进程崩溃」属同一类问题，根因与缓解方式见下文。

---

## 一、日志结论与根因

- **日志路径**：`~/Library/Application Support/Cursor/logs/`（按会话分目录，如 `20260304T190511`）
- **直接表现**：
  1. **渲染进程崩溃（code 5）**：`CodeWindow: renderer process gone (reason: crashed, code: 5)`，随后主进程未捕获异常 `Render frame was disposed before WebFrameMain could be accessed`，导致整个应用退出。
  2. **Agent 分析 SQLite 报错**：`[AgentAnalyticsOperationsMainService] Error ... SQLITE_ERROR: cannot start a transaction within a transaction`（嵌套事务），在「存储 AI code hashes / marking commits as scored」时反复出现，可能加剧不稳定。
  3. **窗口无响应**：`CodeWindow: detected unresponsive` 后有时恢复，有时被关掉。
  4. **ptyHost 心跳超时**：`No ptyHost heartbeat after 6 seconds`，终端相关进程偶发无响应。

- **根因（社区与日志综合）**：
  - **code 5** 在 macOS 上多与 **V8/JS 内存（OOM）**、GC 压力、以及 **macOS mach port 通知失败** 有关，不一定与 GPU 驱动有关；硬件加速已保留，不通过关闭它来规避。
  - **AgentAnalyticsOperationsMainService** 的嵌套事务是 Cursor 端 bug，需等官方修复或通过清理该服务数据缓解。

2. **本机已有设置**

## 四、建议操作（不关功能、从根因缓解）

- **清理缓存（推荐）**：**完全退出 Cursor（Cmd+Q）后**在终端执行以下任一方式，再重新打开 Cursor，可减轻 OOM/旧缓存导致的崩溃：
  - **方式一（一键脚本）**：`bash "/Users/karuo/Documents/个人/卡若AI/运营中枢/参考资料/scripts/clear_cursor_cache.sh"`  
    脚本会检测 Cursor 是否在运行：若在运行会提示先退出再执行；若已退出则自动清理 GPUCache 与 Cache。
  - **方式二（直接命令）**：
  ```bash
  rm -rf ~/Library/Application\ Support/Cursor/GPUCache ~/Library/Application\ Support/Cursor/Cache
  ```
  若 Cursor 正在运行，请先 Cmd+Q 完全退出再执行。
- **向 Cursor 官方反馈**：把闪退时间段的 `main.log`（含 code 5、Render frame disposed、AgentAnalytics SQLITE_ERROR）提交到 [Cursor Forum](https://forum.cursor.com) 或官方支持，便于他们修 code 5 与 SQLite 嵌套事务。
- **保持 Cursor 更新**：当前 2.6.11，后续版本可能修复上述问题。

若官方或社区后续提供「增加 JS 堆大小」等 argv 支持，再在 `argv.json` 中按文档配置，无需关闭硬件加速。

## 九、state.vscdb 为什么这么大？里面装了啥？（3/18 分析）

### 文件结构

`state.vscdb` 是一个 **SQLite 数据库**，路径：  
`~/Library/Application Support/Cursor/User/globalStorage/state.vscdb`  
里面只有 **两张表**：

| 表名 | 行数（约） | value 总大小（约） | 说明 |
|------|------------|--------------------|------|
| **ItemTable** | 2,891 | **~5.5 MB** | VS Code/Cursor 常规全局状态，正常 |
| **cursorDiskKV** | **134.8 万** | **~18.9 GB** | Cursor 自用 KV 存储，**膨胀主因** |

所以 21GB 几乎全是 **cursorDiskKV** 这张表。

---

### cursorDiskKV 里具体是什么？

key 有两种前缀，对应两类数据：

| key 前缀 | 条数（约） | 单条 value 平均大小 | 估算占用 | 含义 |
|----------|------------|----------------------|----------|------|
| **agentKv:blob:** | **60.5 万** | ~14 KB | **~8.3 GB** | Cursor Agent 存的 blob：代码/文件的哈希或嵌入，用于「是否改过」「要不要重算」等，只增不删 |
| **bubbleId:** | **~74 万** | ~13 KB | **~9.6 GB** | 聊天气泡（会话）数据：每个会话一个 bubbleId，下面很多子 key（消息、状态等），历史对话全部持久化 |

- **agentKv:blob**：每个 key 形如 `agentKv:blob:&lt;64 位十六进制&gt;`，value 是二进制 blob（约 14KB/条），和 Agent 分析、codebase 索引、提交评分等有关，没有看到过期或淘汰逻辑。
- **bubbleId**：每个 key 形如 `bubbleId:&lt;会话 UUID&gt;` 或 `bubbleId:&lt;会话 UUID&gt;:&lt;子项 UUID&gt;`，对应 Cursor 里每个聊天会话里的消息和状态，同样只增不删。

---

### 为什么会长到 21GB？

1. **没有淘汰策略**：两类数据都只写不删（或删除很少），时间一长就堆到 134 万条、18.9GB。
2. **Agent 使用频繁**：每次 Agent 分析、索引、评分都会写 `agentKv:blob`，项目多、提交多就会暴增。
3. **聊天会话不清理**：所有历史对话都按 bubbleId 存进 cursorDiskKV，会话越多、对话越长，体积越大。

本质是 **Cursor 把「Agent 缓存」和「聊天历史」都塞进同一个全局 SQLite，且不做容量/时间上限**，所以单文件会涨到 21GB。

---

### ItemTable 里是啥？（正常、可忽略）

约 2,891 条、共约 5.5MB，是常规编辑器状态，例如：

- `Anthropic.claude-code`、`Vue.volar` 等扩展的状态
- `agentLayout.shared.v6`、`adminSettings.cached` 等 Cursor 界面/布局
- `aiCodeTracking.dailyStats.v1.5.&lt;日期&gt;` 按天的统计
- `__$__isNewStorageMarker` 等内部标记

这部分体积很小，**不是** 21GB 的原因。

---

### 小结

| 问题 | 结论 |
|------|------|
| 为什么到 21GB？ | 几乎全是 **cursorDiskKV** 表（约 18.9GB），且 **只增不删**。 |
| 里面装了啥？ | ① **agentKv:blob**：Agent 的代码/文件 blob（约 60 万条、~8.3GB）；② **bubbleId**：所有聊天气泡/消息（约 74 万条、~9.6GB）。 |
| 根因 | Cursor 未对这两类数据做容量或时间淘汰，导致单库无限增长。 |
| 处理办法 | **推荐：`cursor_slim_db.sh` 智能瘦身**（只清缓存，保留对话，VACUUM 压缩）。或用 `cursor_deep_fix.sh` 删库重建（会丢对话）；或用 `cursor_move_global_storage.sh` 迁到其他硬盘。 |

## 十、智能瘦身方案（保留全部对话，3/18）

### 背景

之前的 `cursor_deep_fix.sh` 会删掉整个 state.vscdb，导致所有 Agent 对话历史丢失。
新的 **智能瘦身** 方案只清理 **可再生的缓存**，保留全部聊天对话。

### 本机诊断（3/18）

| 数据 | 行数 | 估算大小 | 是否可清 |
|------|------|----------|----------|
| **agentKv:blob** | 60.6 万 | ~8.3 GB | ✅ 可清（Agent 缓存，Cursor 按需重建） |
| **bubbleId** | 69.8 万（1321 个会话） | ~9.6 GB | ❌ 保留（聊天对话） |
| checkpointId / codeBlockDiff / inlineDiff 等 | 4.9 万 | ~0.5 GB | ✅ 可清（中间态临时数据） |
| **state.vscdb.backup** | — | **15 GB** | ✅ 可删（冗余旧备份） |
| **合计可释放** | | **~24 GB** | |

### 使用方法

**Cmd+Q 退出 Cursor → 终端执行：**

```bash
bash "/Users/karuo/Documents/个人/卡若AI/运营中枢/参考资料/scripts/cursor_slim_db.sh"
```

脚本会：
1. 删除 `agentKv:blob`（Agent 缓存，可再生）
2. 删除中间态临时数据（checkpointId / codeBlockDiff / inlineDiff 等）
3. **VACUUM** 压缩数据库（回收已删数据的磁盘空间）
4. 删除冗余 `state.vscdb.backup`（15GB）

**保留不动的**：所有 `bubbleId`（聊天对话）、`composerData`、`ItemTable`。

### 自动瘦身（已启用）

已注册 macOS LaunchAgent（`com.karuo.cursor-auto-slim`），**每周一凌晨 4:00** 自动执行：
- 仅在 Cursor **未运行时** 才执行（若 Cursor 在运行则跳过）
- 仅在 state.vscdb > 500MB 时才清理
- 日志：`~/Library/Application Support/Cursor/cursor_auto_slim.log`
- 尽量减少同窗口工作区数量（当前 7 个，建议控制在 3 个以内）

## 十、把 state.vscdb 迁到其他硬盘（不影响功能）

**需求**：数据仍保留（不删聊天、不删 Agent 缓存），但希望大文件不要占主盘或想分散 I/O，避免单库在主盘上每次读取过大。

**做法**：把 **整个 globalStorage 目录**（含 state.vscdb 和 Cursor 其它全局状态）**移动到其他硬盘/目录**，在原位置用 **符号链接** 指向新位置。Cursor 仍然只认原路径，读写都走链接，功能完全不变，只是实际文件落在你指定的盘上。

### 步骤（需先退出 Cursor）

1. **Cmd+Q 完全退出 Cursor**。
2. 在终端执行（把 `<目标目录>` 换成你要存放的路径，例如外置硬盘或大容量分区）：
   ```bash
   bash "/Users/karuo/Documents/个人/卡若AI/运营中枢/参考资料/scripts/cursor_move_global_storage.sh" <目标目录>
   ```
3. 示例：
   - 迁到外置硬盘：  
     `bash ".../cursor_move_global_storage.sh" /Volumes/外置硬盘名/CursorGlobalStorage`
   - 迁到用户目录下：  
     `bash ".../cursor_move_global_storage.sh" "$HOME/CursorData/globalStorage"`
4. 脚本会：把当前 `~/Library/Application Support/Cursor/User/globalStorage` **整目录** 移到目标，然后在原位置建符号链接指向新位置。
5. 再打开 Cursor 即可，无需改任何设置。

### 说明

| 项目 | 说明 |
|------|------|
| 是否影响功能 | 不影响。Cursor 仍从原路径访问，只是实际数据在别的盘。 |
| 迁移内容 | **整个 globalStorage**（含 state.vscdb、ItemTable 等），不是单独拆 agentKv / bubbleId（Cursor 不支持的）。 |
| 若目标目录已存在 | 脚本会把已存在的目标目录先备份为 `目标目录.bak.日期时间`，再写入。 |
| 已是符号链接时 | 再次执行会提示「已是符号链接」并显示当前指向，不会重复迁移。 |
| 想恢复回主盘 | 删除原位置的符号链接，把目标目录移回 `~/Library/Application Support/Cursor/User/globalStorage` 即可。 |

### 脚本位置

| 脚本 | 路径 | 用途 |
|------|------|------|
| 迁移到其他硬盘 | `运营中枢/参考资料/scripts/cursor_move_global_storage.sh` | 移动 globalStorage 并建符号链接，不删数据 |
