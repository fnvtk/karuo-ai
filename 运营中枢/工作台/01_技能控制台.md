## 〇、卡若记忆宫殿命名体系

> **正式名称**：**卡若记忆宫殿命名体系**（口述可简称「记忆宫殿命名」）。给每个技能一个**殿→厢→间**式中文锚点，方便卡若口述、Agent 归位；**不替代**五行编号（G/W/M/F/E）与物理目录名，与之**并行**登记。

### 〇.1 结构（殿 → 厢 → 间）

| 层级 | 名称 | 对应关系 | 说明 |
|:---|:---|:---|:---|
| 宫 | **卡若记忆宫殿** | 路径首段 + 体系总称 | 与 **卡若记忆宫殿命名体系** 对应；写入 `memory_palace_path` 时作为第一段 |
| 殿 | **金殿 / 水殿 / 木殿 / 火殿 / 土殿** | 五行 **负责人**（卡资、卡人、卡木、卡火、卡土） | 与 `01_～05_卡X` 目录一致 |
| 厢 | **成员厢** | 掌管人（金仓、水溪、火炬…） | 意象与成员人设一致：**水溪厢** = 清流归档；**金仓厢** = 库藏；**火炬厢** = 光热研发 |
| 间 | **间名（宫殿锚名）** | 单个 SKILL | **2～4 字**为主，可用 **阁、廊、台、池、匣** 等建筑语；须一眼能联想**职能** |

**路径串格式**（写入 `SKILL.md` frontmatter，供人读与官网扩展）：

`卡若记忆宫殿/水殿/水溪厢/回廊洗字`

### 〇.2 命名规则（Agent 分配新人新技能时必须遵守）

1. **先人事，后赐名**：先按职能定 **负责人 → 成员（掌管人）**，再为该技能拟 **间名**；禁止先把名字起得很炫却挂在错误成员下。
2. **人设对齐**：间名气质须与成员 **一句话人设**一致（例：水溪 = 整理归档、清爽；技能名宜 **洗、沉、清、归** 类，不宜 **爆、冲、砸** 类）。
3. **可检索**：`SKILL_REGISTRY` 的「技能」列写 **间名**；若与旧名不同，括号保留 **原名**（如 **回廊洗字**（原名语音转写纠错））。
4. **目录稳定**：**不**因改名批量重命名磁盘文件夹；`legacy_name` 或「原名」字段保留旧称，避免脚本断链。
5. **Frontmatter 建议字段**：`memory_palace_path`（路径串）、`memory_palace_slot`（一句职能锚）；与现有 `name / owner / group` 并存。

### 〇.3 示例（已登记）

| 编号 | 间名（宫殿锚名） | memory_palace_path | 成员 | 人设对齐说明 |
|:---|:---|:---|:---|:---|
| W03b | **回廊洗字** | `卡若记忆宫殿/水殿/水溪厢/回廊洗字` | 水溪 | 口述/ASR 噪声洗净后再入记忆与字幕，贴合「整理归档、清清爽爽」 |

### 〇.4 与登记文件的关系

新技能或改名时：**SKILL.md** 补全 `memory_palace_path` 等字段 → **SKILL_REGISTRY** 技能列 → **本控制台**「按掌管人登记」→ 可选 **skills_export.json** 增加 `memoryPalacePath`（若官网要展示）。

## 一、技能配置说明与参数

### 1.1 SKILL.md 前置元数据（Frontmatter）

卡若AI 内每个技能的 `SKILL.md` 建议包含以下 YAML 头，便于自动化与登记：

| 参数 | 必填 | 说明 |
|:---|:---|:---|
| `name` | 是 | 技能名称，与登记表一致 |
| `description` | 是 | 一句话描述；建议含「触发词：xxx」 |
| `triggers` | 是 | 逗号分隔的触发词，用于 SKILL_REGISTRY 匹配 |
| `owner` | 是 | 掌管人（成员名）：金仓/金盾/水溪/水泉/水桥/木叶/木根/木果/火炬/火锤/火眼/火种/土基/土砖/土渠/土簿 |
| `group` | 建议 | 负责人：金/水/木/火/土 |
| `version` | 建议 | 如 "1.0" |
| `updated` | 建议 | 日期，如 "2026-03-01" |
| `memory_palace_path` | 建议 | 宫殿路径串，如 `卡若记忆宫殿/水殿/水溪厢/回廊洗字`；规则见上节 **〇、卡若记忆宫殿命名体系** |
| `memory_palace_slot` | 建议 | 该技能在宫殿中的**一句职能锚**（便于口述回忆） |
| `legacy_name` | 可选 | 改名前的登记名，供检索与兼容 |

**示例**：

```yaml
---
name: 基因胶囊
description: 将验证过的 Skill 打包为可遗传能力单元。触发词：基因胶囊、打包技能、解包胶囊、继承能力。
group: 土
triggers: 基因胶囊、打包技能、解包胶囊、继承能力、胶囊列表、查胶囊、pack-all
owner: 土砖
version: "1.0"
updated: "2026-03-01"
---
```

### 1.2 登记位置（必填）

凡在卡若AI 内生效的技能，**必须**登记在以下位置，且保持一致：

| 登记项 | 文件/位置 | 说明 |
|:---|:---|:---|
| **技能注册表** | 根目录 `SKILL_REGISTRY.md` | 触发词、路径、成员、一句话；AI 查表路由用 |
| **技能控制台** | 本文件 `运营中枢/工作台/01_技能控制台.md` | 配置/安装/删除/掌管人总览 |
| **能力总索引** | `运营中枢/工作台/00_能力总索引.md` | 按能力类型/操作属性/场景分类（可与注册表互补） |
| **五职能柱·项目矩阵** | `运营中枢/工作台/五负责人_五职能柱_按项目技能矩阵.md` | 五负责人各 5 任务域柱 + 项目 L1/L2/L3 + 技能编号锚点（不重复注册表路径） |
| **开发域模块化索引** | `运营中枢/工作台/开发域_Skill模块化索引.md` | **写代码/搭项目/部署/小程序与获客链路** 按 M1～M11 模块导航；新项目组合速查 |
| **物理路径** | `0X_卡X（X）/成员名_xxx/技能目录/SKILL.md` | 技能必须放在对应「掌管人」目录下 |

**掌管人** = 该技能在 SKILL_REGISTRY 中的「成员」列；一个技能只归属一个成员，该成员所在目录即技能所在路径。

### 1.3 专注清单与番茄钟（卡罗帮 · 仅静默控制）

- **禁止打开图形界面**：不得为「专注清单、轻松清单、WebPomodoro、专注轻探」执行 `open -a WebPomodoro` 或等价操作；**不**让用户在 APP 里点选任务再计时。

## 二、安装流程

### 2.1 卡若AI 内新技能（从零创建）

1. **创建技能**：走「技能工厂」Skill（土砖），用自然语言生成 `SKILL.md` + 目录结构。
2. **放置路径**：将技能目录放到对应**掌管人**目录下，如 `05_卡土（土）/土砖_技能复制/新技能名/SKILL.md`。
3. **登记**：
   - 在 `SKILL_REGISTRY.md` 对应组表下新增一行（#、技能、成员、触发词、SKILL 路径、一句话）。
   - 在本文件「按掌管人登记」中补充（若为新建成员技能则加表）。
   - 可选：在 `00_能力总索引.md` 的完整技能清单/场景中补充。
4. **基因胶囊（可选）**：走「基因胶囊」Skill，`pack` 该技能，便于继承与分发。

### 2.2 从基因胶囊安装（解包）

```bash
cd /Users/karuo/Documents/个人/卡若AI
python3 "05_卡土（土）/土砖_技能复制/基因胶囊/脚本/gene_capsule.py" unpack <胶囊目录或json路径> [-o 目标目录]
```

- 默认按胶囊内 `manifest.skill_path` 写回；`-o` 可指定写入目录。
- 解包后若路径与当前 SKILL_REGISTRY 不一致，需**手动**在 SKILL_REGISTRY 与本控制台补登记或改路径。

### 2.3 Cursor 技能（Agent Skills）

- **卡若自有能力**：正文 **必须** 写在 **`01_～05_卡X` 五行目录**（或 `运营中枢/工作台/` 等已约定位置），并登记 `SKILL_REGISTRY.md` + 本控制台；**禁止**在 **`卡若AI/.cursor/skills/`** 下新增 Skill 正文（该目录仅保留 `README.md` 迁移说明）。
- **第三方 / 纯 Cursor 插件**：可用 Cursor 内 create-skill 或官方文档安装；若后续纳入卡若能力，再按 2.1 **迁入五行目录** 并登记。

### 2.4 Codex 技能

- **安装**：使用 Codex 的 skill-installer（`$CODEX_HOME/skills`），如 `scripts/install-skill-from-github.py --repo <owner>/<repo> --path <path>`。
- **列表**：`scripts/list-skills.py`；安装后需重启 Codex 生效。
- 与卡若AI 登记关系同 2.3：需在卡若AI 内使用时再登记到 SKILL_REGISTRY 与本控制台。

## 三、删除 / 下线流程

### 3.1 从卡若AI 下线某技能

1. **从 SKILL_REGISTRY 移除**：在 `SKILL_REGISTRY.md` 中删除该技能所在行（或整表行）。
2. **从本控制台移除**：在「按掌管人登记」中删除对应技能行。
3. **能力总索引**：在 `00_能力总索引.md` 的完整技能清单/场景中删除或标为「已下线」。
4. **物理文件**：可选保留在成员目录下作归档，或移至 `运营中枢/参考资料/` 下「已下线技能」目录并加说明；若删除目录则需确认无其他引用。

### 3.2 Cursor / Codex 技能

- Cursor：卡若自有技能从 **五行目录** 物理删除或下线，并同步 `SKILL_REGISTRY`；**勿**在 `卡若AI/.cursor/skills/` 留正文副本。
- Codex：从 `$CODEX_HOME/skills` 删除对应技能目录；重启 Codex 生效。

## 五、官网控制台实时同步（卡若AI → 控制台）

- **技能导出文件**：`运营中枢/工作台/skills_export.json`。与 SKILL_REGISTRY 一致，共 64 项；供**卡若ai网站**控制台（`/console/skills`）拉取。
- **更新时间以实际为准**：导出中每条技能可含可选字段 `updated`（ISO 日期时间或 YYYY-MM-DD），与 SKILL.md 的 frontmatter `updated` 或文件修改时间一致；官网控制台「更新时间」列优先显示该值；在控制台编辑保存后也会写回 `updated`，便于下次同步显示实际修改时间。
- **新增技能时**：在 SKILL_REGISTRY 与本文件「按掌管人登记」中登记后，**须同步更新** `skills_export.json`（追加一条 `id, name, member, group, triggers, oneLiner`，可选 `updated`）。之后在官网控制台点击「**从卡若AI同步**」即可实时显示。
- **同步方式**：官网控制台请求 `POST /api/skills/sync`，服务端读取 `skills_export.json`（或环境变量 `KARUO_AI_SKILLS_JSON_PATH` 指定路径）并合并到数据库，控制台列表立即刷新为最新 64 项（或当前导出条数）。
