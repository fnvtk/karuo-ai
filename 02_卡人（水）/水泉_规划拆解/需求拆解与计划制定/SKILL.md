# 需求拆解与计划制定

> 先思考，再拆解，后执行。结合卡火（火）的深度思考与卡人（水）的任务规划能力。

**强制执行前提**（与卡若AI 统一流程一致）：先读全文档与交互流程（总索引、规则、`运营中枢/参考资料/卡若AI交互流程与强制执行条件.md`）→ 搜索扩展执行方案 → 分配任务 → 执行并验证 → **必须拿到目标结果**。

## 核心理念

### 四大原则

1. **思考优先**：每次对话前先深度思考需求本质
2. **拆解为要**：将复杂需求拆解成可执行的原子任务
3. **自动执行**：拆解完成后自动开始执行，逐步推进
4. **持续完成**：理解对话上下文，持续执行直到完成目标

### 协作分工

| 阶段 | 负责人 | 能力 |
|:---|:---|:---|
| **深度思考** | 卡火（火） | Human 3.0追问、CRITIC思考、5 Whys |
| **计划制定** | 卡人（水） | 任务分解、依赖追踪、状态管理 |
| **执行跟踪** | 卡人（水） | 进度更新、风险识别、计划调整 |

## 第一阶段：深度思考（卡火）

### 1.1 每次对话前先思考

**规则**：在开始任何任务前，先进行3-5分钟的深度思考。

**思考框架**：

```markdown
## 🧠 需求思考

### 1. 需求本质（5 Whys）
- 用户真正想要什么？
- 为什么需要这个？
- 不做的后果是什么？

### 2. CRITIC审视
- **C澄清**：需求的具体含义是什么？
- **R假设**：有哪些隐藏假设？
- **I证据**：有数据或案例支持吗？
- **T视角**：不同角色会怎么看？
- **I后果**：做/不做的后果是什么？
- **C原点**：这真的是核心问题吗？

### 3. Human 3.0追问（如需要）
- 目标澄清：解决什么问题？给谁用？
- 边界定义：做什么？不做什么？
- 资源约束：时间？预算？人力？
- 异常处理：失败了怎么办？
- 验收标准：怎么测试？

### 4. 思考结论
- 核心目标：
- 关键约束：
- 风险点：
- 下一步：
```

### 1.2 CRITIC思考模板

| 维度 | 问题 | 输出 |
|:---|:---|:---|
| **C澄清** | "这个需求具体指什么？范围在哪？" | 明确需求边界 |
| **R假设** | "有哪些隐藏假设？这些假设成立吗？" | 识别假设风险 |
| **I证据** | "有数据/案例支持吗？来源可靠吗？" | 验证需求合理性 |
| **T视角** | "用户/开发者/运营会怎么看？" | 多角度审视 |
| **I后果** | "做/不做的短期和长期影响？" | 评估影响 |
| **C原点** | "这真的是最重要的问题吗？" | 回归本质 |

### 1.3 5 Whys深挖技巧

**示例**：

```
用户："我想要一个导出功能"
→ Why 1: 为什么需要导出？
用户："为了做周报"
→ Why 2: 为什么要手动做周报？
用户："因为需要给老板看数据"
→ Why 3: 为什么不能自动生成？
用户："因为数据分散在不同系统"
→ Why 4: 为什么不能统一数据源？
用户："因为历史原因，系统是分散的"
→ Why 5: 为什么不能做数据同步？
→ 本质需求：数据统一 + 自动化报表，而非简单的导出功能
```

## 第二阶段：需求拆解（卡人）

### 2.1 拆解原则

1. **原子化**：每个任务30分钟~4小时可完成
2. **可测试**：每个任务有明确的验收标准
3. **依赖清晰**：明确写出前置任务
4. **优先级明确**：P0核心/P1重要/P2可选

### 2.2 拆解步骤

```
1. 提取功能点
   PRD/需求 → 功能模块 → 功能点

2. 识别依赖关系
   数据库设计 → API开发 → 前端对接 → 测试

3. 设置优先级
   P0: 核心功能，必须完成
   P1: 重要功能，应该完成
   P2: 锦上添花，可以延后

4. 估算时间
   单个任务：30m / 1h / 2h / 4h
   整个项目：按任务累加 + 20%缓冲

5. 生成任务清单
   格式：Epic → Task → Subtask
```

### 2.3 任务粒度标准

| 粒度 | 时间 | 示例 |
|:---|:---|:---|
| **Epic** | 1-2周 | "用户登录模块" |
| **Task** | 4-8小时 | "实现微信登录" |
| **Subtask** | 30分钟-2小时 | "调用微信API获取code" |

## 第四阶段：自动执行（卡人+卡火）

### 4.1 自动执行机制

**核心规则**：拆解完计划后，**自动开始执行第一个任务**，然后逐步推进，直到完成所有任务。

**执行流程**：

```
1. 拆解完成 → 自动识别第一个可执行任务（无依赖或依赖已完成的P0任务）
2. 开始执行 → 标记状态为"进行中" 🔄
3. 执行任务 → 调用相应工具/能力完成任务
4. 更新状态 → 标记为"已完成" ✅，更新执行日志
5. 检查依赖 → 查看是否有任务可以开始（依赖已完成）
6. 执行下一个 → 自动执行下一个可执行任务
7. 循环执行 → 重复步骤2-6，直到所有任务完成
8. 完成确认 → 检查成功标准，确认目标达成
```

### 4.2 上下文理解

**规则**：每次执行前，必须理解对话上下文：

1. **读取对话历史**
   - 理解用户之前说了什么
   - 识别需求的变化和补充
   - 结合之前的拆解结果

2. **理解当前任务**
   - 当前要执行的任务是什么？
   - 这个任务在整体计划中的位置？
   - 需要什么资源/信息？

3. **结合上下文执行**
   - 不要孤立地执行任务
   - 考虑前后任务的关联
   - 保持执行的一致性

### 4.3 执行原则

1. **自动开始**：拆解完成后立即开始执行第一个任务
2. **逐步推进**：完成一个任务后自动执行下一个
3. **及时更新**：每完成一个任务立即更新状态和日志
4. **持续思考**：遇到问题时重新思考（调用卡火）
5. **灵活调整**：根据实际情况调整计划
6. **直到完成**：持续执行直到所有任务完成或用户明确停止

### 4.4 状态流转

```
⏳待开始 → 🔄进行中 → ✅已完成
    │           │
    │           └→ 🚫阻塞中 → 🔄进行中
    │
    └→ ❌已取消

自动执行触发条件：
- 任务状态为"⏳待开始"
- 所有依赖任务已完成
- 优先级为P0或P1
- 自动开始执行，无需用户确认
```

### 4.5 阻塞处理

当任务遇到阻塞时：

1. **识别阻塞原因**
   - 技术问题？
   - 依赖未完成？
   - 需求不明确？
   - 资源不足？

2. **自动重新思考**（调用卡火）
   - 用CRITIC审视问题
   - 寻找替代方案
   - 评估影响
   - 记录思考过程

3. **自动调整计划**（卡人）
   - 更新任务状态为"🚫阻塞中"
   - 调整依赖关系
   - 重新安排优先级
   - 寻找可替代路径

4. **继续执行**
   - 如果找到替代方案，继续执行
   - 如果无法解决，等待用户输入
   - 记录阻塞原因和解决方案

### 4.6 任务执行循环

**自动执行循环**：

```python
def execute_plan():
    plan = load_plan()  # 加载需求拆解计划
    
    while not plan.is_complete():
        # 1. 找到下一个可执行任务
        next_task = plan.find_next_executable_task()
        
        if next_task is None:
            # 检查是否有阻塞任务
            blocked_tasks = plan.get_blocked_tasks()
            if blocked_tasks:
                # 尝试解决阻塞
                resolve_blocked_tasks(blocked_tasks)
            else:
                # 所有任务完成
                break
        
        # 2. 理解上下文
        context = understand_context(next_task, conversation_history)
        
        # 3. 执行任务
        result = execute_task(next_task, context)
        
        # 4. 更新状态
        plan.update_task_status(next_task, result)
        
        # 5. 更新日志
        plan.log_execution(next_task, result)
        
        # 6. 检查是否需要用户确认
        if result.requires_user_confirmation():
            wait_for_user_input()
    
    # 7. 完成确认
    verify_success_criteria(plan)
    return plan
```
